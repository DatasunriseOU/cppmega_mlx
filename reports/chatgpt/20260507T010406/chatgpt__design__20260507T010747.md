# Design & Architecture Review – Round‑1 mlx-z3-wiring

## HIGH Priority

### 1. Tight coupling of Z3 logic into MLX kernels
**Files:** `_msl_transform.py:72-160`, `fp8_vecmat_path_c.py:45-110`  
- The Z3 `PassConfigs` and `ConstraintScope` logic is embedded directly into kernel setup functions.  
- **Issue:** This creates a strong dependency on Z3 within otherwise generic kernel code, violating separation of concerns. Any change in Z3 API would require touching MLX kernel files.  
- **Suggestion:** Introduce a `Z3Interface` abstraction or adapter layer that handles all Z3-related setup and proofs. MLX kernels should only call high-level methods like `z3.verify_constraints(buffer)` rather than managing scopes directly.

### 2. Redundant kernel-specific checks mixed with assertions
**Files:** `fp8_vecmat_path_c.py:88-102`, `topk_selector.py:33-47`  
- Checks like `K % 4` are retained as assertions but still scattered in kernel initialization code.  
- **Issue:** Mixing validation logic into kernel dispatch code increases cognitive load and reduces readability. Assertions here double as implicit contracts.  
- **Suggestion:** Centralize validation in a separate configuration/validation module and make kernel code purely operational.

---

## MEDIUM Priority

### 3. Lack of modularity for feature gates
**Files:** `_msl_transform.py:180-220`, `sparse_mla.py:51-87`  
- `PassConfigs` and MLX kernel modes are configured inline without a clear registry or plugin pattern.  
- **Issue:** Adding new kernel modes or tuning passes would require edits to multiple files.  
- **Suggestion:** Introduce a registry pattern for MLX passes (e.g., `PassRegistry.register("fp8_vecmat", FP8PassConfig)`) to decouple kernel implementation from configuration and enable easier extension.

### 4. Duplicated slicing/tiling logic
**Files:** `_msl_transform.py:110-135`, `sparse_mla.py:72-82`  
- Several functions perform almost identical buffer tiling/slicing operations with slight differences for FP8 vs int32 kernels.  
- **Issue:** This violates DRY principle and makes future optimizations harder.  
- **Suggestion:** Consolidate tiling logic into a utility function with parameters for type, stride, and alignment, then call from both FP8 and sparse kernels.

### 5. Weak naming for kernel parameters
**Files:** `fp8_vecmat_path_c.py:58-66`, `topk_selector.py:20-33`  
- Variables like `buf_a`, `buf_b`, `K`, `N`, `M` are used extensively without context.  
- **Issue:** Makes code less self-documenting and increases risk of misconfigurations.  
- **Suggestion:** Use descriptive names like `input_buffer`, `weight_matrix`, `num_features`, `num_vectors` to improve clarity and reduce cognitive load.

## LOW Priority

### 6. Inconsistent docstrings / inline comments
**Files:** `_msl_transform.py:40-70`, `sparse_mla.py:10-25`  
- Some functions have detailed docstrings; others rely on inline comments only.  
- **Issue:** Inconsistent documentation reduces maintainability, especially for a complex system like MLX-Z3 integration.  
- **Suggestion:** Standardize docstring style across all kernel files and include expected input shapes, types, and preconditions.

### 7. Implicit assumptions about buffer alignment
**Files:** `fp8_vecmat_path_c.py:92-100`, `sparse_mla.py:64-70`  
- Kernel code assumes buffers are aligned to 16/32-byte boundaries without enforcing it at the interface.  
- **Issue:** Violates encapsulation; callers must know alignment rules.  
- **Suggestion:** Enforce alignment checks at kernel entry or through typed wrapper classes (e.g., `AlignedBuffer`) to make assumptions explicit.

---

### Summary

- **Primary design risk:** Strong coupling of Z3 logic directly into kernel dispatch and tight inline checks.
- **Secondary concerns:** Duplicated tiling logic, weak naming, and missing plugin/registry pattern for MLX passes.
- **Actionable steps:**  
  1. Introduce a `Z3Interface` or adapter.  
  2. Centralize validation logic.  
  3. Refactor tiling/slicing into reusable utilities.  
  4. Implement a registry for MLX pass configurations.  
  5. Improve naming and documentation consistency.
