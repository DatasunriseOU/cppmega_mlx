### **1. Top-K Path C: Insertion-sort fix + perf regression**

**What likely went wrong, design-wise**  
The bug `early-break corrupted insertion sort` and `1.28x slower for K=32` suggests the abstraction was leaking.

**Design issues to check:**
1. **Violated abstraction: `topk_kernel.py`**  
   If Path C exposes raw insertion-sort details to the caller, you’ve coupled algorithm choice to API. The `AUTO` fallback to Path B means the caller can’t rely on Path C semantics.  
   **Fix**: Hide all paths behind `topk(x, k, method='auto')`. Path selection should be internal. If `method='path_c'` can silently fail perf, that’s a leaky abstraction. Add a contract: either guarantee Path C semantics+perf, or remove it.

2. **Missing extension point: kernel selection**  
   `AUTO routes Path B now` implies hard-coded routing. If you add Path D later, you’ll edit multiple places.  
   **Fix**: Strategy pattern. `TOPK_STRATEGIES = {32: PathB, 64: PathC}` with a registry. `topk.py:42` should just dispatch. Add cost model hook: `strategy.estimate_cost(shape, k, dtype)` for future auto-tuning.

3. **Over-engineering: hand-rolled insertion sort in Python/MLX**  
   If `topk_insertion_sort` is pure Python for `K=16/32`, you’re fighting the framework. MLX `argpartition` + `take` is 1 op.  
   **Fix**: Simplify. Delete custom Path C unless you can prove >20% win on M-series with `mx.compile`. Otherwise you maintain buggy code for -28% perf.  
   Cite target: `kernels/topk_path_c.py:1-80` if that file exists — candidate for deletion.

4. **Duplicated logic: early-break guards**  
   The `early-break` bug usually happens when you copy the same loop guard in `forward` and `backward` or in `compile` vs `eager`.  
   **Fix**: Single source of truth. Put the invariant `assert heap[0] <= new_val or break` in one helper. Test it.

### **2. `sparse_mla canonicalize` + MSL compile hoist issues**

**Design issues to check:**
1. **Leaky module: `sparse_mla.py` knowing about MSL hoisting**  
   If `canonicalize` has `if mx.in_compilation():` or special cases for `mx.compile`, you’ve coupled a math op to a compiler quirk.  
   **Fix**: `canonicalize` should be pure math. Hoist-safety is a lowering concern. Move MSL workarounds to `mlx/lowering/sparse_mla.msl`. `sparse_mla.py:120` shouldn’t mention `compile`. 

2. **Fragile extension point: `_path_c_rewrite_merge_round`**  
   Your TODO says it’s fragile. That’s code smell for “implicit protocol”.  
   **Fix**: Make rewrite rules explicit. If `merge_round` expects tensors in canonical order, enforce with `assert is_canonical(x)` at entry, not a TODO. Add `SparseMLATensor` wrapper that carries `canonical: bool` in metadata. `rewrite_merge_round.py:55` becomes a method: `x.to_canonical().merge_round()`.

3. **Awkward coupling: sparse layout + attention logic**  
   If `sparse_mla` calls directly into `topk` Path C, you’ve coupled a high-level op to a low-level kernel choice.  
   **Fix**: Dependency inversion. `sparse_mla(..., topk_fn=topk)` with default. Lets you unit test with fake topk and swap kernels without touching MLA code.

### **3. Cross-cutting: Naming, clarity, over/under-engineering**

1. **Naming: `Path B`, `Path C`, `_path_c_rewrite_merge_round`**  
   Paths are meaningless to new devs. `c_rewrite` suggests C code, but this is MLX.  
   **Fix**: `topk_heap_kernel`, `topk_bitonic_kernel`, `mla_merge_canonical`. Names should describe algorithm, not git history.

2. **Under-engineered: No perf guardrails**  
   You shipped 1.94x speedup that was wrong. That means no correctness benchmark gated the optimization.  
   **Fix**: `tests/test_topk.py`: add `pytest.mark.parametrize("k", [16,32,64])` + reference `torch.topk`. Fail CI if `max_diff > 1e-5`. Add `pytest-benchmark` and fail if regression >10% vs main.

3. **Over-engineered: 3 paths for topk**  
   If K<1024, MLX `argpartition` is already GPU-friendly. Custom kernels only win for `K≈4096` on long seq.  
   **Fix**: Delete Path C. Keep 1 fast path + fallback. Complexity budget: each kernel = 2x maintenance cost.

4. **Missing abstraction: `mlx-z3-wiring`**  
   The name suggests this repo wires MLX to Z3 solver. If `topk` and `sparse_mla` are in same package, that’s leaky.  
   **Fix**: Split: `mlx_z3.core` = pure MLX ops, `mlx_z3.z3` = constraint logic. Don’t let Z3 bugs break topk.
