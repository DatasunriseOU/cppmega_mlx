### 1. TopK insertion-sort fix – correctness vs security
**File: `topk_selector.py:995-1014`**

```python
995 | if value > local_vals[0]:
996 | pos = 0
997 | # Insertion sort into the ascending top-K list...
1001 | # fix-round-3: the prior `elif (K<=8) or (K>=64): break`
1002 | # was a tuning heuristic that *skipped* the break for
1003 | # K in {16, 32}, leaving the loop walking past the
1004 | # insertion point and corrupting later shifts
1007 | for p in T.serial(1, _TOPK_C_K):
1008 | if value > local_vals[p]:
1009 | local_vals[p - 1] = local_vals[p]
1010 | local_idx[p - 1] = local_idx[p]
1011 | pos = p
1012 | else:
1013 | break
1014 | local_vals[pos] = value
1015 | local_idx[pos] = j
```

**Security assessment**:
1. **Fixed bug = removed attacker primitive**: The buggy early-break for K=16/32 allowed the loop to run past `p = _TOPK_C_K-1`, writing `local_vals[p]` where `p == K`. Because `local_vals` is statically sized `[K]`, that was an OOB write in threadgroup memory. In Metal, OOB writes to threadgroup memory corrupt neighbor thread data or can fault the kernel. An attacker controlling `scores` could craft values to trigger the corrupted path and DoS the GPU or leak cross-batch data. **Fix removes this**. Current code has correct bounds: `T.serial(1, K)` never accesses index `K`.
2. **No taint from shapes**: `K` comes from `_TOPK_C_K`, which is set from Python arg `k` in `_path_c_kernel_for`. There is validation: `topk_selector.py:372-374` raises if `k < 1 or k > seq_len`. No integer overflow: `k` is `int` from Python, checked before use. No injection into MSL string — K is baked as literal `TOPK_C_K` in the macro, not string-concat of user data.
3. **Remaining risk: threadgroup memory exhaustion**: K is template param. Host code doesn’t cap K except by `shared_mem_size` check `topk_selector.py:514-516`. If attacker can request K=8192, `merge_vals[K_DOUBLE]` = 16KB floats = 128KB + indices = 192KB threadgroup, exceeds M1’s 32KB limit. Kernel build would fail closed. Not exploitable, but DoS if exposed via API. **Recommendation**: Add hard cap `k <= 256` or `k <= THREADS` in `topk_selector()` before kernel build.

**Verdict**: (1) Correct now. No other corruptions visible in diff. The merge tree below `1018-1044` uses power-of-two `stride` and bounds checks `if lane < stride`. No OOB.

### 2. Sparse-MLA hoist fix – MSL codegen tampering risk
**File: `sparse_mla_path_c.py:124-159`**

```python
124 | def _strip_z3_hoisted_address_decls(msl: str) -> str:
125 | """Drop z3-final's algebraic hoists..."""
144 | hoisted_names = ("h", "b", "g", "q_row_base", "kv_b_base",...)
156 | hoisted_pattern = re.compile(
157 | r"(?m)^[ \t]*int (?:h|b|g|q_row_base|...) = "
158 | r"(?:0|\([^\\n]*(?:threadgroup_position_in_grid\.x|gid)[^\\n]*\));\n"
159 | )
160 | return hoisted_pattern.sub("", msl)
```

**Security assessment**:
1. **Regex injection surface**: This function does string surgery on compiler-emitted MSL. The regex is anchored and only deletes `int <name> =...;` lines where `<name>` is in allowlist. It cannot delete arbitrary code. Pattern cannot be tricked by user strings because `msl` comes from TileLang, not user input. **No injection**.
2. **Logic bug = potential undef behavior**: If the regex fails to strip a hoisted decl, `_canonicalize_fwd_base_indexing` later redeclares same var as `uint`. MSL disallows redeclaration → compile error, fail-closed. If it over-strips, you get use-of-uninit. Both are compile-time or GPU faults, not memory corruption. Attacker can’t control Z3 output, so not exploitable.
3. **No new MSL compile issues seen**: The hoist strip runs before `_canonicalize_fwd_lane_indexing`, which injects `int gid`. Ordering is correct. Grep shows no other `gid` decls before that point. **Hoist-aware fix looks safe**.

**Residual risk**: Any code that does `re.sub` on generated code is brittle. If TileLang changes var names, strip fails open. Add a compile-time `static_assert` that `gid` isn’t declared twice: parse MSL for `r"\bint\s+gid\b"` before/after. Fail fast.

### 3. Other Path C optimizations – “fast because broken”
Scanned for similar perf hacks:

**File: `sparse_mla_path_c.py:82-100` `_remove_redundant_kv_bounds_checks`**
Strips `if (0 <= gather_idx && gather_idx < seq_len_kv)`.

**Security assessment**:
1. **Removes runtime bounds check**: If `gather_idx` comes from `indices[b][k]`, and attacker controls `indices`, OOB read on `kv` buffer. Metal OOB reads return 0 or fault. This is only safe if upstream guarantees `indices < seq_len_kv`. In `sparse_mla` Python wrapper, `indices` is masked by `starts/ends` but not clamped. **Potential SSBO OOB read if caller passes bad indices**.
2. **Mitigation exists**: `sparse_mla.py` reference path clamps via `mx.where`. Path B/C assume valid indices. If this module is exposed as public API, add assert: `assert mx.all(indices < seq_len_kv)`. Current code trusts caller. Document this precondition or enforce it.
3. **Not new in this diff**, but perf gain comes from deleting safety. Flag as authz/trust-boundary issue: kernel assumes caller validated indices.

**File: `sparse_mla_path_c.py:102-122` `_insert_forward_all_masked_fast_return`**
Adds early return when `row_max == -INFINITY`. No security impact.

### Concrete attacker scenarios

1. **GPU DoS via huge K**: Call `topk_selector(scores, k=4096)`. Passes Python check `k <= seq_len`, but `K_DOUBLE=8192` blows threadgroup memory → Metal compile error, aborts process. If this is in a service, repeat to crash it. Fix: `if k > 512: raise ValueError`.
2. **OOB read via bad indices**: Call `sparse_mla_path_c.forward(q, k, v, indices=full_of_SEQ_LEN)`. Bounds check was stripped by `_remove_redundant_kv_bounds_checks`. GPU reads past `kv` buffer. On Apple GPU this usually returns 0, but spec allows fault. If fault, DoS. If 0, correctness issue only. Fix: add `indices = mx.minimum(indices, seq_len_kv-1)` in Python wrapper.

### Performance notes, since you asked

1. **TopK**: After fix, Path C is 1.28x slower than Path B for K=32. Expected: insertion sort is O(K) per element, worse than radix. For K>64, consider disabling Path C or switching to bitonic. Current code already AUTO-routes Path B – correct.
2. **Sparse-MLA**: Hoist strip is O(len(msl)) regex. Fine for <100KB kernels. If TileLang emits >1MB MSL, compile time spikes. Cache lowered MSL by `(batch, seq_len, k, d_v, threads)` key; you already `@lru_cache` the kernel.
