### 2. Remaining / new security issues

#### **2.1 Integer overflow in threadgroup allocation → OOB write, DoS**

**Location**: `dsa_splitk_indexer_loss.py:605`  
`_MD_PRE_BYTES = 8 * AH * BLOCK_SQ`

**Vulnerability**: `AH`, `BLOCK_SQ` are template ints from caller. No validation. If attacker controls model config and sets `AH=2**20`, `BLOCK_SQ=2**20`, product overflows 32-bit int in TileLang/TVM lowering, wraps to small positive. `USE_MD_PRE` becomes `True` incorrectly.

**Attacker scenario**: Supply malicious checkpoint with `num_heads=1048576`. Kernel compiles, `T.alloc_fragment((AH, BLOCK_SQ), "float32")` under-allocates, subsequent `M_pre[hh, i] = M[b, hh, sq_idx]` writes OOB. On Metal this corrupts threadgroup memory → wrong gradients, crash, or info leak via side channel.

**Fix**: Add compile-time assert:  
```python
if AH <= 0 or AH > 1024 or BLOCK_SQ <= 0 or BLOCK_SQ > 256: 
    raise ValueError("AH/BLOCK_SQ out of safe range")
_MD_PRE_BYTES = 8 * int(AH) * int(BLOCK_SQ)  # force 64-bit
```

#### **2.2 Missing validation on `Sk` before `T.alloc_shared` → DoS via huge alloc**

**Location**: `dsa_splitk_indexer_loss.py:279-281`  
`K_shared = T.alloc_shared((BLOCK_SK, BLOCK_D), "float16")`  
`BLOCK_SK` derived from `Sk` via tiling. If caller passes `Sk=2**30`, Metal kernel launch requests `BLOCK_SK*BLOCK_D*sizeof(half)` threadgroup memory >32KB. 

**Attacker scenario**: Call `dsa_splitk_indexer_loss_tilelang` with `K.size(2)=1<<30`. Metal driver rejects kernel → abort. If host doesn’t check return code, it’s a DoS. No explicit `Sk < MAX` check.

**Fix**: Validate at wrapper entry:  
```python
if Sk > 65536: raise ValueError("Sk too large for Metal threadgroup")
```

#### **2.3 Top-K sentinel `-1` not filtered in downstream scatter → OOB read**

**Location**: `topk_selector.py:336, 551, 555`  
Kernel returns `-1` for invalid slots.  
`dsa_splitk_indexer_loss.py:1074-1077` does `.scatter_(-1, topk_idx64, 0.0)` after clamping. 

**Vulnerability**: Clamp sets OOB → `-1`, but `scatter_` with index `-1` on last dim is legal in PyTorch: writes to last element. If attacker crafts `starts/ends` so `valid_counts=0`, all `topk_idx64` become `-1`, scatter writes `0.0` to `Index_Mask[..., -1]` every call, corrupting mask for legitimate positions.

**Attacker scenario**: Data poisoning: feed batch where one sample has `starts[b]=ends[b]`. That row’s `Index_Mask` gets `-1` column zeroed. Later sparse attention uses wrong mask → model ignores real tokens, degrades accuracy for targeted input.

**Fix**: After clamp, also mask scatter: `topk_idx64 = torch.where(topk_idx64>=0, topk_idx64, topk_idx64.size(-1))` then scatter, or skip rows with `valid_counts==0`.

#### **2.4 No check for `Index_Scores` dtype/stride → type confusion / info leak**

**Location**: `dsa_splitk_indexer_loss.py:1032-1037`  
Wrapper accepts any `torch.Tensor` for `Index_Scores`. If attacker passes `dtype=int64` tensor that aliases memory of a secrets buffer, kernel `T.Buffer` interprets bits as `float32` without cast.

**Attacker scenario**: In multi-tenant inference server, tenant A passes `Index_Scores` that is actually a `torch.frombuffer` view of tenant B’s API key. Kernel reads it, computes `log(softmax)`, differences leak via timing/gradient. No `dtype==torch.float32` assert before launch.

**Fix**: `assert Index_Scores.dtype in (torch.float16, torch.bfloat16, torch.float32)` before TileLang call.

#### **2.5 Path traversal via `os.environ` in fp8_vecmat_path_c**

**Location**: `fp8_vecmat_path_c.py:125-130`  
`if os.environ.get("CPPMEGA_FP8_VECMAT_PATH_C_NO_Z3", "0") not in ("0", "", "false", "False"): return {}`

**Vulnerability**: Not a direct traversal, but env var controls Z3 PassConfig. Attacker with env control can disable `tl.drop_provable_bound_checks`. That pass removes `if (i < N)` guards. Disabling it keeps redundant checks, but if future commit relies on the pass for safety, toggling env re-introduces OOB. Defense-in-depth violation.

**Fix**: Don’t gate security-critical passes on env. Hard-code `drop_provable_bound_checks=True`.

#### **2.6 Race condition: shared `FP8_VECMAT_PATH_C_PASS_CONFIGS_CACHE`**

**Location**: `fp8_vecmat_path_c.py:133-138`  
Global cache populated under `RLock`. Good. But `topk_selector.py` has similar pattern at `L161-L166` with `threading.Lock()` not `RLock`. 

**Vulnerability**: If a reentrant call occurs during cache init, deadlock. Not auth-sensitive, but DoS in server. Auth-sensitive if kernel build hangs during login request.

**Fix**: Use `RLock` consistently, or `threading.local()` for per-thread config.

### 3. Performance notes that impact security

1. **HBM fallback path**: When `USE_MD_PRE=False`, every `sk_tile` reloads `M/D` from HBM. Side-channel risk: memory bandwidth pattern leaks `AH`. If `AH` is secret (e.g., per-user MoE), attacker co-located on GPU can infer it. Keep budget gate, but add constant-time option.

2. **Metal `atomic_max` on fp32**: Implemented as CAS loop. Under contention, latency spikes → timing side channel. Not critical unless `amax` used for gating.

### 4. Concrete recommendations

1. **Add input sanitizers at wrapper boundary** for all kernels:
   ```python
   def _check(t, name, max_dim=1<<20):
       if not t.is_cuda and not t.device.type=="mps": raise ValueError
       if any(s <= 0 or s > max_dim for s in t.shape): raise ValueError(f"{name} dim OOB")
   ```
2. **Sentinel handling**: Replace `-1` with `torch.iinfo(torch.int32).min` and assert after topk: `assert (topk_idx >= 0).all() or ignore_row`.
3. **Remove env toggles for safety passes**. Use build-time flags.
4. **Fuzz test**: Generate `AH` in `[1, 2, 4, ..., 2**20]`, `Sk` in `[0, 2**30]` and ensure no crash. Current code will Metal-compile-fail at `AH>64` but passes CUDA.
