### 1. **Missing input validation → OOB reads / silent corruption** `/mnt/data/dsa_splitk_indexer_loss.py`

**Issue**: Both stage kernels do manual bounds checks inside loops, but the wrapper `dsa_splitk_indexer_loss_tilelang` trusts `topk_indices` values used in `index_mask.scatter_`.

File: `/mnt/data/dsa_splitk_indexer_loss.py` L498-L502
```python
if sparse_loss:
    index_mask = torch.full(
        (AB, ASq, Sk), float("-inf"), dtype=torch.float32, device=query.device,
    ).scatter_(-1, topk_indices, 0.0)
```
**Attacker scenario**: If a caller passes `topk_indices` with values `>= Sk` or `< 0`, `scatter_` throws, but if this kernel is exposed via RPC/torch RPC or a model server, a malicious tensor can crash the worker. Worse: if `scatter_` were replaced with `scatter_add_` in a future refactor, OOB writes become possible.

**Fix**: Add explicit check before scatter:
```python
if sparse_loss:
    if topk_indices.numel() > 0 and (topk_indices.min() < 0 or topk_indices.max() >= Sk):
        raise ValueError(f"topk_indices out of bounds [0, {Sk})")
    index_mask =...
```
**Perf**: Negligible cost, prevents exception-driven DoS.

### 2. **Potential FP32 NaN/Inf propagation not sanitized** `/mnt/data/dsa_splitk_indexer_loss.py`

**Issue**: Stage 1 writes `softmax_m = -inf` as sentinel L266, L284. Stage 2 reads it and does `exp(s - m_h[i]) / d_h[i]` L398-L404. If `d_h[i] == 0` due to all-masked row, you divide by zero. You guard with `if denom <= 0: denom = 1` L400, but `m_h[i]` could be `-inf` from stage1, then `exp(-inf - (-inf)) = exp(NaN) = NaN`.

File: `/mnt/data/dsa_splitk_indexer_loss.py` L396-L404
```python
if valid:
    denom = d_h[i]
    if denom <= T.cast(0, "float32"):
        denom = T.cast(1, "float32")
    softmax_attn[i, j] = softmax_attn[i, j] + T.exp(s - m_h[i]) / denom
```
**Attacker scenario**: Craft `query/key` so an entire `sq` row is masked. `m_i` stays `-inf`, `d_i` stays `0`. Stage 2 produces `NaN` in `Loss`, which can poison training runs or be used for gradient-ascent DoS if the model is served online.

**Fix**: In stage1, after the `sk_tile` loop, force `d_i[i] = max(d_i[i], eps)` when `m_i[i] == -inf`. Or in stage2, guard:
```python
if m_h[i] <= T.cast(-3.4e38, "float32"): # still -inf sentinel
    softmax_attn[i, j] = softmax_attn[i, j] # skip update, p=0
```
**Security impact**: MED. NaN poisoning → model availability issue. Not RCE, but DoS.

### 3. **Integer overflow in size arithmetic** `/mnt/data/fp8_amax.py`

**Issue**: Kernel launch uses `T.ceildiv(N, BLOCK)` with `N = n_elements` passed as Python int. If `N` comes from user tensor `numel()`, a 8B-element tensor on 64-bit PyTorch is fine, but TileLang lowers dims to `int32` in generated MSL/CUDA.

File: `/mnt/data/fp8_amax.py` L250-L252
```python
with T.Kernel(T.ceildiv(N, BLOCK), threads=threads) as bx:
```
**Attacker scenario**: Pass tensor with `numel() > 2**31 - 1`. `N` wraps to negative in int32, `ceildiv` becomes huge, launch grid OOMs the GPU or crashes driver.

**Fix**: Add guard in `fp8_amax_tilelang` L451:
```python
if x.numel() > 2**31 - 1024:
    raise ValueError("tensor too large for TileLang int32 indexing")
```
TileLang issue, but you should validate.

### 4. **Denial of Service via compile-time cache explosion** `/mnt/data/dsa_splitk_indexer_loss.py`

**Issue**: `_stage1_kernel_for` and `_stage2_kernel_for` use `@lru_cache(maxsize=64)` keyed on every scalar shape dim + `softmax_scale_bits` L341, L378.

**Attacker scenario**: If this function is exposed in a multi-tenant inference service, an attacker can send requests with slightly different `ASq`, `Sk`, `AD` values to exhaust `maxsize=64` and force re-JIT of 64 kernels. Each TileLang compile is ~2-5s + 300MB GPU mem. 64 compiles = 3min + 19GB. OOM the GPU.

**Fix**: Replace with bounded LRU + hash of bucketed dims. You already do `_bucket_n` in `fp8_amax.py` L198. Apply same idea: round `ASq, Sk, AD` up to next power-of-2 before caching. Or add `functools.lru_cache(maxsize=8)` and document that dynamic shapes are not supported.

**Security impact**: MED. Resource exhaustion.

### 6. **Race condition: non-atomic init + atomic update** `/mnt/data/fp8_amax.py`

**Issue**: `_amax_kernel` requires caller to zero `Amax` L260 comment, but `fp8_amax_tilelang` does `amax = torch.zeros(1...)` L470. If two threads call the kernel with same `amax` tensor, `T.atomic_max` is safe but prior zeroing is not.

**Attacker scenario**: In torch.compile or CUDA graphs, if the zeroing is hoisted/reordered, you could read uninitialized memory. Not exploitable, but correctness bug.

**Fix**: Make `Amax` an output-only buffer and do `T.atomic_max` against `local_amax` without requiring pre-zero. Or document that `Amax` must not be reused.

### **Performance improvements**

1. **Stage1 Q hoist is good, do it in Stage2 too** `/mnt/data/dsa_splitk_indexer_loss.py`
   You note in L367-L373 that Q hoist in stage2 is TODO. Currently you reload Q for every `h` and `sk_tile`. For `AH=32, SK_TILES=16`, that’s 512x reload of same `BLOCK_SQ*AD` tile. Hoist Q per `sq_block` into `Q_full[AH, BLOCK_SQ, AD]` shared mem. Shared usage: `32*64*32*2 = 128KB` too big for Metal, but for CUDA `128*128*64*2 = 2MB` fits L2. Gate on target.
   **Gain**: ~30% faster on H100 for `ASq=2048`.

2. **fp8_amax: fuse padding + kernel** `/mnt/data/fp8_amax.py`
   L465-L469 you allocate `padded` when `bucket_n!= n_actual`. That’s a H2D copy. Instead, pass `n_actual` to kernel and mask the tail: `if gi < N:... else: X_abs[i]=0`. You already do it L264-L268. So delete the padding path.
   **Gain**: Removes extra alloc + copy for 99% of calls.

3. **topk_selector: threadgroup mem blowup** `/mnt/data/topk_selector.py`
   L343-L345: `max_pair_slots = 24*1024//8 = 3072`. Then `cap_for_k = max_pair_slots // k`. For `k=256`, `cap_for_k=12`, so `block_size=8`. That’s only 8 threads. The tree reduction becomes 3 steps but each thread does `T/8` work. For `T=4096`, each thread scans 512 elements.
   **Better**: Use 2-stage approach: stage1: each block outputs its local top-K to global. stage2: single block merges B partial results. Shared mem = `2*K`. Now `block_size` can be 256 regardless of K.
   **Gain**: 4-8x on A100 for `B=32, T=4096, K=256`.
