Got it — I dug through all 4 Path C kernels. Overall the codebase is well-documented and the TileLang→CUDA/Metal duality is ambitious. But there are several design/architecture issues that make it fragile and over-complicated. No HIGH/MED blockers from Wave-1, but here's the cross-check:

### **1. Repeated "Path C availability" abstraction is duplicated 4x**

**Issue**: Every file re-implements the same pattern: `_tilelang_available()`, `*PathCStatus` dataclass, `*_path_c_status()`, plus copy-pasted docstrings.

**Evidence**: 
- `dsa_splitk_indexer_loss.py:170-199`
- `fp8_amax.py:149-177`
- `fp8_vecmat_path_c.py:144-279`
- `topk_selector.py:697-727`

**Design problem**: Violated DRY + leaky abstraction. Each kernel decides its own "am I available?" logic, but the real dependency is `tilelang + tvm + metal backend`. Callers can't reason about Path C as a platform capability.

**Simplify to**: Single `cppmega_mlx.nn._tilelang.status.path_c_available() -> bool, str`. Kernel modules just call it. Add per-kernel overrides only if a kernel has extra deps. Right now you have 4 near-identical `_tilelang_available()` impls that could drift.

### **2. PassConfig filtering logic is copy-pasted with different names**

**Issue**: Same Z3 PassConfig probing pattern in 3 files, different function names.

**Evidence**:
- `fp8_vecmat_path_c.py:76-109` `_filter_supported_pass_configs`
- `topk_selector.py:168-186` `_topk_filter_supported_pass_configs`
- `fp8_vecmat_path_c.py:124-144` uses `_fp8_vecmat_path_c_pass_configs_cache_lock`

**Design problem**: Duplicated logic + awkward coupling. The "which PassConfigs exist in this TVM build" is a platform query, not kernel-specific. Lock per file means 3 caches.

**Simplify to**: `tilelang/utils.py::get_active_pass_configs()` with single LRU cache. Kernels just import it. The comment `Idea #10 is NOT a PassConfig` appears verbatim 2x — that's a code smell.

### **3. Metal vs CUDA shared memory budget leaks into kernel source**

**Issue**: BLOCK_* constants and memory math are scattered through PrimFunc bodies with `if "metal" in target` comments. The abstraction "TileLang handles both backends" leaks.

**Evidence**:
- `dsa_splitk_indexer_loss.py:132-146` `_metal_block_overrides()` with 32KB math in docstring
- `dsa_splitk_indexer_loss.py:362-363` `# Metal 32*64*2 = 4 KB` inline comment
- `dsa_splitk_indexer_loss.py:574` `# (CUDA 128*128*4 = 64 KB worst case; Metal 128*32*4 = 16 KB)`
- `topk_selector.py:736-740` `max_shared_bytes = 32 * 1024`

**Design problem**: Violated abstraction. Kernel DSL shouldn't know Apple's 32KB limit. Every new kernel repeats this calculation. If Apple ships 64KB threadgroup memory, you grep all files.

**Simplify to**: Target policy object. `get_target_policy("metal") -> {shared_mem_budget: 32768, warp_size: 32}`. Kernels call `policy.shared_mem_per_block()` and let TileLang schedule pick BLOCK_*. Remove `BLOCK_SQ/BLOCK_D` from source; make them autotuned args. The `T.alloc_shared` sizing becomes the compiler's job.

### **4. Path B vs Path C naming/clarity problem**

**Issue**: "Path B" and "Path C" appear in comments everywhere but aren't defined in code. New devs can't tell which is which.

**Evidence**: 
- `fp8_vecmat_path_c.py:4-5` "Path C ... counterpart to the hand-written Path B MSL"
- `topk_selector.py:52-53` "Path B skips TileLang entirely and emits MSL by hand"
- `topk_selector.py:340` comment "Direct-MSL Path B kernel"

**Design problem**: Implicit taxonomy. Path B = hand-written `mx.fast.metal_kernel`. Path C = TileLang DSL. But the files are named `*_path_c.py` while containing Path B fallbacks.

**Fix**: Central `ARCHITECTURE.md`. In code: rename to `fp8_vecmat_tl.py` + `fp8_vecmat_msl.py`. Or use a registry: `@kernel_impl(backend="tilelang")` vs `@kernel_impl(backend="msl")`. The `backend: "auto"` param in `topk_selector.py:1157-1161` becomes the only place that knows both paths.

### **5. Over-engineered status dataclasses**

**Issue**: `DSASplitKPathCStatus`, `FP8AmaxPathCStatus`, `FP8VecmatPathCStatus` are all identical: `{available, reason}`.

**Evidence**: Lines 170-178, 149-157, 144-153 across files

**Design problem**: Over-engineering. You don't need a class per kernel. A `namedtuple` or `dict` works. These dataclasses are never serialized or extended. They exist only for `._replace(reason=...)` which `str` would do.

**Simplify to**: Return `tuple[bool, str]` or raise `PathCUnavailable(reason)`. Saves 30 LOC/file.

### **6. Shared memory allocation strategy is inconsistent**

**Issue**: Some kernels use `T.alloc_shared`, others comment that it fails on Metal and fall back to direct MSL. The abstraction leaks.

**Evidence**:
- `dsa_splitk_indexer_loss.py:328-329` uses `T.alloc_shared`
- `topk_selector.py:43-45` "T.alloc_shared ... lowers to storage scope shared.dyn which TVM's Metal codegen rejects"
- `topk_selector.py:936-944` `T.alloc_shared(..., scope="shared")` in Path C anyway

**Design problem**: Awkward coupling. `topk_selector.py` says `T.alloc_shared` is broken on Metal, but `dsa_splitk_indexer_loss.py` uses it. Either TileLang fixed it, or topk is outdated, or DSA will break. Callers can't trust the abstraction.

**Action**: Pick one: If TileLang 0.1.9+ supports `scope="shared"` on Metal, delete the `shared.dyn` comments in `topk_selector.py:44-45`. If not, gate all `T.alloc_shared` behind `if target=="cuda"`. Don't mix.

### **7. Missing extension point for fused kernels**

**Issue**: `fp8_amax.py:38-41` says "Fused amax + quantize -- the current pipeline keeps the Triton two-pass shape". Same for `dsa_splitk_indexer_loss.py:74-75` mentions Phase 2.4 for fp8 variants.

**Design problem**: No extension point. If you want fused amax+quantize later, you rewrite both kernels. The PrimFunc signatures are hardcoded.

**Simplify to**: Design PrimFuncs to take optional `out_buffers` or use `T.call_extern` chaining. Better: expose `tilelang.jit(compile_only=True)` and let a Python orchestrator fuse graphs. Right now each file is a dead-end kernel, not a composable op.

### **8. Threading locks per file for global state**

**Issue**: Each file has its own `_lock = threading.Lock()` for PassConfig cache.

**Evidence**: `fp8_vecmat_path_c.py:73`, `topk_selector.py:165`

**Design problem**: Over-engineering. PassConfig probing is idempotent and cheap. 3 locks for 1 global TVM state. If TileLang isn't thread-safe during import, you already lost.

**Simplify to**: One lock in `tilelang/utils.py`. Or delete locks — cache population races don't matter if the result is deterministic.
