Why

  • On current HEAD, full Path C backward improves BF16 fast rows versus the safe forward-C/backward-B route, but still loses to Path B on the main fast 8-bit acceptance rows.
  • The remaining BF16 loss is not explained by isolated Mamba3 or M2RNN kernels; the full train graph still pays composition overhead across generated TileLang/tvm-ffi surfaces.
  • Path B uses one established Metal dispatch family for BF16 recurrence rows, while Path C composes path_c_tilelang_dsl_fwd_path_b_bwd plus path_c_tilelang_dsl_packed_post; FP8 adds tilelang_fp8_prepared_path_c_fwd.
  • FP8 Path C active memory stays essentially equal to Path B, but peak memory rises by roughly 21-32 GiB on most rows, pointing at temporary/high-water lifetime around prepared FP8 buffers and graph specialization, not retained model tensors.
  • Warm Path C first steps are still high, so cache reuse is incomplete; this is a codegen/cache-key and graph-specialization target, separate from steady-state kernel speed.
  • The fp8/lion Path B baseline changed between e3926e8 and current f49a7e4; current recheck confirms the slow baseline, so it is marked unstable and should not be used as evidence that Path C generally wins.
