Why Path C Loses

Evidence-backed classification

Primary loss: Path C loses on fast optimizer rows where kernel/graph overhead is exposed: BF16 adam8bit/lion8bit and FP8 adam8bit/lion/lion8bit are all around 0.65-0.67x C/B. Active memory is unchanged, so this is not live tensor growth.

  • Mamba3 is not the current kernel loser: focused BF16 1B bench has Path C fwd+bwd at about 0.556x Path B time.
  • M2RNN is not the main isolated loser: focused BF16 split bench has fwd+bwd C/B about 0.988x.
  • Composition is the problem: in full train-step, Path C adds generated-kernel graph boundaries and compile/cache overhead. On fast optimizer rows this adds roughly +0.9s to +1.2s steady step.
  • FP8 has an additional memory/high-water problem: active memory matches Path B, but peak memory rises by about +21 to +32 GiB, pointing at temporary buffers/compile/graph lifetime, not retained model tensors.
  • Warm cache is incomplete: many Path C warm rows still have first-step cost from 12s to 45s; cache metadata says hit, but train graph specialization is still happening.
