Path C Cache Delta Memory Profiler

bf16 AdamW, local_gb10_quarter, batch=1, seq=2048, grad checkpoint, one profiled training step. Generated 2026-05-17.

Answer

The +10.9 GiB cache delta is not needed for correctness. It is retained MLX Metal allocator cache: freed IOGPUMetalBuffer resources kept for reuse. mx.clear_cache drops the Path C cache to 0 after each step, while active memory stays at the same 17.09 GiB.

Profiler Setup

  • xctrace Allocations was attempted first, but on this host it hung before Python startup in liboainject.dylib and then failed attach; that trace was not used as evidence.
  • The usable profiler run used MallocStackLogging, a live vmmap snapshot during an in-process hold, and malloc_history for allocation stacks.
  • A profiling-only --profile-hold-seconds flag was added to scripts/m04_train_step.py so external profilers can inspect the live MLX cache before process exit.

The peak high-water does not decrease after the fact because it records the maximum allocation reached earlier. The post-run cache decreases to zero with mx.clear_cache, and a new Path C local_gb10 default now applies mx.set_cache_limit(0) before model allocation so this cache is not retained after the step.

Conclusion

  • The cache is useful only as a speed/fragmentation tradeoff for reusing Metal buffers. It is not required for correctness.
  • For reports, cache delta must be shown separately from active memory and must not be described as model memory growth.
  • For Path C local_gb10 runs, the runtime now sets mx.set_cache_limit(0) by default; override with CPPMEGA_MLX_CACHE_LIMIT_BYTES or --cache-limit-bytes when measuring cache reuse deliberately.
