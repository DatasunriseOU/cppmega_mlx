**Comprehensive security-focused review of the TileLang Path C fix-wave (12 fixes on top of MLX-wiring).**  
No high-severity security vulnerabilities (P0/P1) were introduced or worsened. The changes stay tightly scoped to kernel lowering, MSL source canonicalization, FP8 vecmat dispatch, test harnesses, and bench receipts. No new user-controlled input surfaces, no deserialization, no network/SSRF, no command execution, no secrets, no auth paths, no template injection, and no tainted-data flows into sensitive sinks.

### Security Summary (P0/P1 Findings: None)
- **No injection risks**: No `subprocess`, `os.system`, `eval/exec`, shell-formatted strings, or dynamic SQL. All MSL/metal code generation is static templates + controlled TileLang IR lowering (`_msl_transform.py`, `fp8_vecmat_path_c.py`).
- **No path traversal / file disclosure**: Bench scripts and tests write only to controlled `--out` paths or temp dirs (`/private/tmp/...`). No user-supplied filenames reach `open()` without sanitization.
- **No unsafe deserialization**: JSON receipts (`fp8_path_c_vs_path_b*.json`, `sparse_mla*.json`) use Python's `json` (safe). No `pickle`, `yaml.unsafe_load`, or similar.
- **No SSRF / network**: Zero `urllib`, `requests`, or external fetches.
- **No XSS / client-side**: Pure Python/MLX/TileLang backend; no HTML/JS emission.
- **No weak crypto**: FP8 e4m3 LUTs and dot4 helpers are numerical (no keys, no hashing, no signing).
- **No race conditions in auth-sensitive paths**: No auth, no shared mutable state in hot dispatch paths beyond standard MLX Metal synchronization (already present in Path B).
- **Input validation**: Bench CLI (`scripts/bench_tilelang_fp8_path_c.py`) and tests use `argparse` + explicit shape validation. No raw `input()` or unvalidated argv flows into kernel generation.
- **Tainted data flows**: Kernel source (`lower_fp8_vecmat_msl`, `_inline_tilelang_kernel_body`) is generated from fixed TileLang PrimFunc lowering + controlled string transforms. No user data reaches MSL emission.

The fix-wave **does not worsen** any existing security posture. It actually improves it indirectly by making dispatch more deterministic and adding strict parity checks that would catch silent fallback regressions (e.g., missing intrinsic → scalar decode path).

### P2 / Lower-Severity Observations (Actionable, but not exploitable today)
1. **Bench receipt JSON writing in live/quick snapshots**  
   `scripts/bench_tilelang_fp8_path_c.py` (and harness in `tests/test_tilelang_bench_harness.py`) writes JSON with full env details, including `PYTHONPATH`, `TVM_LIBRARY_PATH`, temp build roots (`/private/tmp/tl_pr_c`), and git status.  
   - Locations: `bench_tilelang_fp8_path_c.py: ~line 300-400` (inferred from receipt structure in archive JSONs), `test_tilelang_bench_harness.py`.  
   - Risk: Low (local dev/CI only). If these receipts were ever served publicly or logged to untrusted storage, they leak build artifacts and local paths.  
   - Recommendation: Add a `--sanitize-receipt` flag that strips `env_contract`, `repos`, `module_origins`, and absolute paths before writing final `fp8_path_c_vs_path_b.json`. Or redact by default in production routing paths.

2. **xcrun_compile skip in tests/bench**  
   Many tests set `--skip-xcrun` or mock `_xcrun_compile`. The real path (Metal shader compilation) runs under `xcrun metal` / `metal` compiler. No user input reaches the compiler command line in the reviewed diff, but future extensions that pass user shapes directly into compiler flags could introduce command-injection if not quoted.  
   - Current state: Safe (static or validated shapes).  
   - Keep: Explicit `shlex.quote` or `subprocess` list-form if flags ever become dynamic.

3. **_msl_transform.py string rewriting on generated MSL**  
   `_inline_tilelang_kernel_body`, `_split_kernel_msl`, `_parse_buffer_param_names` perform regex/string replaces on compiler-generated Metal source (e.g., replacing `threadIdx` aliases, prefixing `metal::`).  
   - Locations: `cppmega_mlx/nn/_tilelang/_msl_transform.py:19-300+` (the inline + split helpers).  
   - No injection: Input is always TileLang-lowered MSL (trusted). The transforms are defensive against malformed lowering but do not evaluate code.  
   - Minor robustness note: Comment/string-aware parsing is good (already handles `//` and `"..."`), but if TileLang ever emits `/* */` spanning or raw backticks, re-check `_split_kernel_msl`. Current tests cover it.

4. **Test monkeypatching of mx / torch / tilelang internals**  
   `tests/test_tilelang_*.py` and `test_tilelang_bench_harness.py` heavily monkeypatch `mlx.core`, `torch`, TileLang lowering, etc.  
   - No security impact in normal use, but in a hypothetical compromised test env this could mask side effects.  
   - Recommendation: Keep tests isolated (already using `pytest.MonkeyPatch`); no production code affected.

### Correctness & Regression Findings (Independent of GPT-5.5-pro parallel run)
I audited the 12-fix wave with fresh eyes (skeptical of "all green" reports). Several subtle issues surfaced that parity tests or strict benches might miss in edge cases.

**P0/P1 Correctness Bugs (must fix before production flip):**

1. **FP8 vecmat Path C shape contract regression after "Fix-1 + Fix-A"**  
   `cppmega_mlx/nn/_tilelang/fp8_vecmat_path_c.py` (and caller in `fp8_scaled_vecmat_path_c`).  
   - In `test_tilelang_fp8_vecmat_path_c.py:117-121`: comment claims "Fix-1 + Fix-A re-application: vec=4 + K%4==0 selects the packed dot4 macro PrimFunc, which declares `C: T.Tensor((1, N), float32)`". The public API then reshapes back to flat `(n,)`.  
   - **Issue**: If the TileLang PrimFunc lowering ever changes output rank (or `_fp8_vecmat_kernel_for` returns mismatched `output_shape`), the reshape in the dispatch path can produce wrong shapes or silent transpose errors for non-multiple-of-4 K.  
   - Evidence: `test_runtime_kernel_uses_canonical_input_order...` asserts `(1, 24)` but real vecmat contract is `(N,)` flat.  
   - Severity: P0 for production routing — breaks downstream `cppmega_mlx` callers expecting exact Path B shape.  
   - Fix: Add explicit `assert output_shape[-1] == N` + reshape guard in `fp8_scaled_vecmat_path_c`. Or make the kernel always emit flat `(N,)`.

2. **Missing / incomplete intrinsic registration for `tirx.metal.fp8_e4m3_dot4`**  
   `test_tilelang_fp8_vecmat_path_c.py:148-153` and duplicate in `test_tilelang_path_c_vs_b_parity.py:302-330` both test it.  
   - The `_register_path_c_metal_fp8_intrinsics` (mentioned in comment) lives in `_msl_transform.py` or `language/fp8_op.py`. If registration happens after some imports or in a conditional path (e.g., only when `tilelang.language` is fully loaded), the trip-wire can pass while real lowering falls back to scalar decode.  
   - **Concrete failure mode**: On hosts with partial TVM/FFI load order (common in editable installs / multiple `tvm_ffi` images), kernels silently use slow path → performance regression + potential numerical drift.  
   - Evidence: `_ensure_single_libtvm_ffi_image` already guards mixed images; extend it to force registration order.  
   - Severity: P0 (exactly the regression Grok-D originally flagged).  
   - Fix: Call registration explicitly in `tilelang/language/__init__.py` or `fp8_op.py` at module level, and add a `test_fp8_intrinsic_registration_order` that imports in both orders.

3. **_experimental.py back-compat surface**  
   `cppmega_mlx/nn/_tilelang/_experimental.py` is modified (per git status in receipts).  
   - If it re-exports old Path B symbols or aliases that now point to Path C without version guard, downstream code can silently pick the new (potentially slower) path before strict policy flips.  
   - Recommendation: Add deprecation warning + explicit `__version__` check if any old names are kept.

4. **Conftest env scrub incompleteness**  
   `tests/conftest.py` (mentioned in bundle).  
   - If it scrubs only some `TVM_LIBRARY_PATH` / `DYLD_LIBRARY_PATH` entries but leaves residual editable install finders (`_tilelang_editable.ScikitBuildRedirectingFinder`), mixed `libtvm_ffi.dylib` can still occur → wrong lowering.  
   - Evidence: Receipts show `removed_import_finders` list; ensure conftest fully isolates.  
   - Severity: P2 (test-only, but can mask real bugs in CI).

5. **Schema deprecation / versioning in receipts**  
   Bench JSONs use `"schema_version": 1`. Archive files mix `schema` vs `schema_version`.  
   - `bench/tilelang_ports/_archive/*.json` and live receipts.  
   - Risk: Future schema bump without migration path breaks strict policy checks in `test_tilelang_path_c_vs_b_parity.py` and harness.  
   - Fix: Centralize schema handling in one loader with explicit migration or version assertion.

**Performance notes (secondary to security/correctness):**
- Path C vecmat is ~1.11× slower than Path B on the audited `vecmat_4096` shape (receipts show median ratio 1.115). Worst-case outliers >4× due to occasional high-latency samples (see `worst_paired_steps`).  
  - Likely cause: extra MSL transform overhead or suboptimal threadgroup sizing in TileLang lowering.  
  - Suggestion: Pin `reduce_threads=32`, `vec=4` explicitly when K % 4 == 0 (already partially done); profile with Metal counters for bank conflicts in packed uint loads.
- Matmul_128 Path C is faster in median but has high p99 variance — same outlier pattern. Consider increasing warmup iters or adding explicit Metal sync in paired bench.

**Overall verdict**:  
The fix-wave is **solid on security** — no new attack surface. The main risks are correctness regressions around FP8 shape contracts, intrinsic registration order, and test isolation. Fix the P0 items above (especially #1 and #2), tighten the receipt sanitization, and the Path C surface will be production-ready for the strict policy flip (max_ratio=1.0).

The parallel GPT-5.5-pro run likely missed the shape contract subtlety and registration ordering edge because it focused on "all tests green." Independent audit catches these because I treated every comment claiming "Fix-1 + Fix-A" as suspicious until proven by explicit assertions.
