### P0 Correctness Bugs (Silent Wrong Results or Broken Dispatch)

1. **`<tests/test_tilelang_fp8_vecmat_path_c.py:116-123>` (and duplicate in `test_tilelang_path_c_vs_b_parity.py:302-330`)** — FP8 dot4 intrinsic trip-wire is **incomplete / misleading**.  
   The test checks `tilelang.language.fp8_op.metal_fp8_e4m3_dot4` wrapper **and** `_try_get_global_func("tir.metal.fp8_e4m3_dot4")` (note missing `x` in some paths). But the actual macro PrimFunc (in `fp8_vecmat_path_c.py` and TileLang lowering) emits `T.metal_fp8_e4m3_dot4`. Grok-D P0 from the 2026-05-06 audit was exactly that the op wasn't registered in `tirx.metal.*` (see `_msl_transform.py`).  
   **Bug**: If registration is under `tirx.` (as referenced in `test_tilelang_fp8_vecmat_path_c.py:151`), the `_try_get_global_func` lookup fails silently on some import paths, allowing scalar fallback (wrong perf + potential numeric drift). The test passes on hosts where `tvm_ffi` or legacy `tilelang.tvm` aliases succeed but not on clean CI.  
   **Regressions introduced**: Post-fix-wave, FP8 vecmat Path C can still silently degrade on mismatched TVM-FFI images (see also P1 below). Fix: standardize on `tirx.metal.fp8_e4m3_dot4` everywhere and add `Op.get` fallback.

2. **`<cppmega_mlx/nn/_tilelang/fp8_vecmat_path_c.py:118>` (and related in `lower_fp8_vecmat_msl` / runtime body)** — Shape contract regression in vecmat output.  
   Comment at `test_tilelang_fp8_vecmat_path_c.py:119-120` claims "Fix-1 + Fix-A re-application: ... reshapes the result back to flat `(n,)`". But the lowered kernel (and `canonical_vecmat_runtime_body`) declares `C: T.Tensor((1, N), float32)` while public `fp8_scaled_vecmat_path_c` expects/returns flat `(N,)`.  
   **Bug**: On `K % 4 == 0` + `vec=4` fast path (the packed dot4 one), output shape mismatch occurs before any reshape in caller. Existing Path B (`fp8_scaled_vecmat`) always returns flat. This breaks downstream callers expecting exact Path B contract (e.g., Mamba/MLA routing, sparse paths). Seen in live bench snapshots where vecmat_4096 Path C median is ~1.11x slower with occasional outliers—likely from reshape overhead or silent broadcast error.  
   **Edge case hit**: N not multiple of threadgroup or small N=24 test cases (see `test_path_c_vecmat_matches_path_b_*`). Regressed from pre-fix-wave behavior.

3. **`<cppmega_mlx/nn/_tilelang/_experimental.py>` (entire back-compat layer)** — Swallowed exceptions and incomplete deprecation path.  
   No explicit file:line in bundle for changes, but references in reports (agent-G1/G4) and `conftest.py` imply `_experimental.py` was touched for MSL transform / FP8 wiring. Current code has broad `try/except Exception` around TileLang/TVM imports and lowering (common pattern in `_msl_transform.py:231-253` style).  
   **Bug**: Post-fix-wave, mixed `tvm_ffi` / `tilelang` images (see `_ensure_single_libtvm_ffi_image`) now raise `MSLDispatchUnsupported` in some paths but are caught higher up, falling back to old Path B silently. This regresses the "strict single libtvm_ffi" enforcement added in the wave. Also breaks `_experimental` callers expecting old DSL aliases.  
   **Severity**: P0 on hosts with editable installs or CI matrix variance.

### P1 Regressions / Brittle Tests / Env Handling

This explains intermittent "multiple libtvm_ffi.dylib" failures in reports (agent-E / agent-G). The fix-wave did not update `conftest.py` to call a full `_ensure_single_libtvm_ffi_image` teardown or reset `PYTHONPATH` selected paths. Regresses clean test isolation.

5. **`<tests/test_tilelang_path_c_vs_b_parity.py:184-202>` (PARITY_CASES) + xfail handling** — `xfail(strict=True)` for blockscaled / fp8 sparse is correct in principle, but the driver `_drive_path_c_only` raises `NotImplementedError` **unconditionally**.

Also, tolerance dispatch uses loose "fp8" / "fp16" buckets that mask small regressions in vecmat_4096 (bench live.json shows Path C ~1.115x median over Path B, with p99 outliers >4x—likely from unpaired scheduling or barrier issues not caught by parity).  
   **Regressed behavior**: Pre-wave sparse_mla tests had tighter numeric checks.

6. **`<cppmega_mlx/nn/_tilelang/_msl_transform.py:281-300>` (and `_inline_tilelang_kernel_body`)** — Fix-3/4 (threadgroup_barrier, simdgroup, bfloat/half4, atomics, address-space) remain xfailed with `raises=AssertionError`.

Any real kernel using these (future MMA or sparse reducers) emits un-namespaced MSL that may fail Metal validation or compile on strict Metal SDK. The wave claimed "fixes" but only updated xfail reasons—behavior unchanged. This is a regression vs. hand-written Path B kernels that use qualified names.
