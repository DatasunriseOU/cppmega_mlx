**Comprehensive Design & Architecture Review: TileLang Path C Fix-Wave (post-ebbb84b on MLX wiring)**

The bundle shows a mature, high-stakes integration of **TileLang DSL kernels** (Path C) into the **cppmega_mlx** stack, with **Path B** (hand-written MSL vendored kernels) as the performance/compatibility baseline. The 12-fix wave (Grok + GPT agents) cleaned up correctness, tests, and benches, but several **architectural smells** remain. I prioritize **violated abstractions**, **leaky coupling**, **duplicated logic**, **naming/clarity**, **extension-point gaps**, and **schema/compat debt**.

I am skeptical of "it works on my Mac Studio" assumptions—Metal lowering quirks, TVM/FFI dylib loading, and MLX dispatch surfaces create many silent fallback paths. The review focuses on design, not micro-benchmarks (though vecmat regression in live.json is noted).

### 1. P0/P1 Correctness & Intrinsic Registration Bugs (Critical for FP8 Path C)

**`cppmega_mlx/nn/_tilelang/_msl_transform.py` (and related)**

- **P0: Duplicate/weak FP8 dot4 intrinsic registration trip-wire.**  
  `tests/test_tilelang_path_c_vs_b_parity.py:302-330` and `tests/test_tilelang_fp8_vecmat_path_c.py:125-154` both implement nearly identical `_try_get_global_func` + `Op.get("tir.metal.fp8_e4m3_dot4")` checks. This is **duplicated logic** and **leaky abstraction**—the registration belongs in `_msl_transform.py:_register_path_c_metal_fp8_intrinsics` (or a dedicated `tilelang/language/fp8_op.py` init).  
  If registration fails silently (as in Grok-D P0), both Path C FP8 kernels (vecmat + any future matmul) fall back to scalar decode → **massive perf cliff + correctness risk on non-dev hosts**. The tests catch it only when TVM is importable.  
  *Recommendation:* Move registration + single robust `is_fp8_dot4_registered()` helper to `_msl_transform.py:xx` (near `_ensure_single_libtvm_ffi_image`). Make both tests import and call it. Simplify the multi-FFI probing.

- **P1: `_experimental.py` back-compat surface is leaky and under-tested.**  
  The file is referenced in module origins and reports but not deeply exercised in the bundle's tests. If it re-exports or monkey-patches lowering paths (common in TileLang experimental), it risks **violating the Path C vs Path B parity contract**. No dedicated test for experimental → Path C changes can regress without CI failure.  
  *Action:* Either delete/deprecate `_experimental.py` or add a parity test in `test_tilelang_path_c_vs_b_parity.py` that exercises it explicitly.

### 2. Shape Contract Regressions & Vecmat Path C (High Impact)

**`cppmega_mlx/nn/_tilelang/fp8_vecmat_path_c.py`**

- **P1: Output shape contract fragility in `_fp8_vecmat_kernel_for`.**  
  Lines ~108-122 (test reference): After "Fix-1 + Fix-A", the macro PrimFunc returns `(1, N)` but the public `fp8_scaled_vecmat_path_c` reshapes back to flat `(N,)`. This is **leaky abstraction**—callers (MLX dispatch, bench harness, routing) must know about the internal reshape.  
  The test asserts `output_shape == (1, 24)` but the public API contract is flat. This creates **coupling** between TileLang lowering and MLX runtime dispatch.  
  *Fix (simplify):* Make the TileLang kernel emit the public flat shape directly, or encapsulate reshape inside a thin wrapper that also handles `None` (unavailable) case uniformly. Document the contract in the function docstring with explicit "always returns (N,) or None".

- **P2: Per-row vs scalar scale handling duplication.**  
  `canonical_vecmat_runtime_body` has explicit branches for `scale_w_per_row`. Features inspection (`fp8_vecmat_msl_features`) and blockers repeat similar checks. This is minor but indicates the **runtime body** is becoming a grab-bag.  
  *Suggestion:* Extract a small `VecmatScaleMode` enum or dataclass to centralize the logic and make extension (e.g., per-column scales) obvious.

**Bench/JSON evidence:** In `fp8_path_c_vs_path_b.current_strict.json` (and live), vecmat Path C is ~1.11–1.12× slower than Path B on median, with severe tail latency (p99 ~4×). Matmul_128 Path C is actually faster in some snapshots. This suggests **the vecmat lowering is not yet Path-B-competitive**—design should expose tunable parameters (vec, reduce_threads, outputs_per_block) via the status object for the router to pick optimally.

### 3. Architecture & Abstraction Violations (Core Design Issues)

**Routing & Dispatch (`docs/production_kernel_routing.md`, `_msl_transform.py`, sparse_*_path_c.py files)**

- **Leaky "Path C status" abstraction across modules.**  
  Every `_path_c.py` file (fp8_vecmat_path_c.py:25-28, sparse_mla_path_c.py, sparse_mla_blockscaled_path_c.py:28-41, mamba3_path_c.py) defines its own `XXXPathCStatus` / `XXXQKReducePathCStatus` + `features()` + `blockers()` + `lower_XXX_msl()`.  
  This is **massive duplication** and **violated single-responsibility**. The status objects mix "is available?", "why not?", "shape params", "generated MSL features", and "strict policy hints".  
  *Strong recommendation (simplify):* Introduce a **single `TileLangMSLKernelStatus`** base in `_msl_transform.py` (or a new `cppmega_mlx/nn/_tilelang/status.py`). Subclasses only override domain-specific fields. Centralize `msl_features(msl_text: str) -> dict` and `blockers(...)` there. This makes the AUTO router trivial and prevents drift (e.g., sparse blockscaled vs fp8 vecmat using different marker keys).

- **Missing extension point for new kernels.**  
  Adding a new Path C kernel (e.g., the planned m2rnn_path_c) requires touching: bench script, test parity sweep, routing doc, strict policy in receipt JSON, and multiple status helpers. No registry or plugin mechanism.  
  *Fix:* A simple `KERNEL_REGISTRY: dict[str, Callable[[...], TileLangMSLKernelStatus]]` in `_tilelang/__init__.py` or a dedicated router module. Each kernel module registers itself on import.

**Test Architecture (`tests/test_tilelang_path_c_vs_b_parity.py`, `test_tilelang_*.py`)**

- **Over-reliance on strict-xfail for missing surfaces.**  
  `test_tilelang_path_c_vs_b_parity.py:216-219`, `test_tilelang_sparse_mla_blockscaled_path_c.py:183-189`, `test_tilelang_m2rnn_path_c.py:69-76` etc. use `pytest.xfail(strict=True)` for "not implemented yet". This is clever for flip-to-green, but **pollutes the test suite** and makes "green CI" ambiguous (how many xfails are expected?).  
  *Better design:* Use `pytest.importorskip` + a top-level `PATH_C_IMPLEMENTED = {"sparse_mla": True, "mamba3": True, ...}` constant (or auto-discover via registry). Mark whole modules or parametrized cases cleanly.

- **conftest.py env scrubbing incompleteness.**  
  The bundle mentions `tests/conftest.py` but the snippet in reports suggests it may not fully isolate TVM_LIBRARY_PATH / DYLD_LIBRARY_PATH / multiple libtvm_ffi.dylib images (see `_ensure_single_libtvm_ffi_image`). This is a **P1 correctness risk** for CI vs local dev divergence.  
  *Action:* Audit `conftest.py` to force a clean single-image environment + document the contract.

**Schema & Receipt Design (`bench/tilelang_ports/*.json`, tests)**

- **Schema_version + deprecation path missing.**  
  Receipts use `"schema_version": 1` (or "schema": 1 in sparse). No forward-compat or migration hooks. When Path C full fwd/bwd lands, old receipts will break strict checks.  
  *Suggestion:* Add `"schema_version": 2` with explicit migration in the bench harness loader, or use a versioned strict_policy object.

- **Duplicated strict policy logic.**  
  `fp8_bench._shape_row_strict_ok`, `sparse_bench._strict_row_failures`, and receipt JSON policies repeat similar ratio/parity checks. Centralize in a `StrictPolicy` dataclass with `validate_row(row)`.

### 4. Naming, Clarity & Over/Under-Engineering

- **Inconsistent naming across Path C modules:**  
  `fp8_scaled_vecmat_path_c` (good) vs `sparse_mla_path_c_apply` vs `mamba3_mimo_apply_path_c` vs `blockscaled_sparse_mla_qk_reduce_path_c`.  
  *Standardize:* `xxx_path_c.apply(...)` or `xxx_path_c_dispatch(...)` for uniformity. The `_path_c` suffix is good; keep it.

- **"Path B fast path ready" flag in blockers** (`fp8_vecmat_path_c.py:984-1009` etc.):  
  This is a **leaky coupling** to the router. Better: return a `dispatch_recommendation: Literal["path_b", "path_c", "hybrid"]` or let the status expose a `preferred_path` + confidence.

- **_msl_transform.py is becoming a god module.**  
  It handles inline rewriting, splitting, dispatch, FFI image checks, and (soon) registration. Split into `_msl_rewriter.py`, `_msl_dispatch.py`, `_ffi_safety.py` if it grows further.

### 5. Performance & Router Design Suggestions (Simplify)

The live/current_strict benches show Path C vecmat **regressing** vs Path B while matmul sometimes wins. The router (production_kernel_routing.md) should be **data-driven** on the status objects rather than hard-coded.

- **Simplest win:** Make every status object expose `estimated_tflops` or `median_ms_proxy` (from a tiny probe run) so the router can pick the faster one automatically, falling back to Path B only on known blockers. This eliminates manual ratio thresholds in strict_policy.

- **Avoid over-engineering:** Do **not** add full auto-tuning yet. Keep TileLang lowering parameters (vec, threads, stages) exposed via status, but let a thin router decide based on shape + features.

### Summary of Actionable Priorities (P0 → P2)

**P0 (must fix before next merge):**
- Centralize FP8 dot4 registration + single test helper (`_msl_transform.py`).
- Fix vecmat output shape contract leak.

**P1 (design hygiene):**
- Unify status abstraction + blockers/features.
- Clean up xfail proliferation with registry or importorskip.
- Audit `_experimental.py` and conftest env isolation.

**P2 (nice-to-have):**
- Standardize apply naming.
- Add schema migration story.
- Make router status-driven instead of ratio-hardcoded.

The fix-wave improved correctness surface area significantly, but the **proliferation of per-kernel status/helpers** and **leaky TileLang→MSL→MLX dispatch layers** are the biggest remaining architectural debts. Addressing the status unification will make adding future kernels (m2rnn, full sparse fwd/bwd, etc.) trivial and keep Path C vs Path B parity enforceable.

The code is otherwise solid—strong test coverage via parametrized parity, good use of strict policies, and clear separation of lowering vs runtime. With the above cleanups, Path C can become the default without risk.
