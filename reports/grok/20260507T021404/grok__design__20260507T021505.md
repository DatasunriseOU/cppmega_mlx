**Design and architecture review of the attached TileLang Path C modules** (`dsa_splitk_indexer_loss.py`, `fp8_amax.py`, `fp8_vecmat_path_c.py`, `topk_selector.py`). Focus is on **violated abstractions**, **leaky modules**, **duplicated logic**, **awkward coupling**, **naming/clarity**, **extension points**, and **over-/under-engineering**. All references use exact file:line numbers from the provided sources (as of the attachment). Suggestions prioritize simplification while preserving the unified CUDA/Metal goal and conservative Z3 wiring (PassConfig opt-in via per-module caches with `threading.Lock`, defaults OFF except fp8 dot4 env-gated and proven barrier elision).

### 1. Violated Abstractions & Leaky Modules (Core Issue)
The modules repeatedly violate the **TileLang DSL abstraction** (tiles as first-class, high-level dataflow with `T.Kernel` / `alloc_shared` / `alloc_fragment` / `T.gemm` / `T.Pipelined` / `T.reduce_*`) by:
- Mixing low-level manual index arithmetic, bounds predicates, and sentinel handling (`-3.4028234663852886e38`, `T.infinity`) inside the `@T.prim_func`.
- Duplicating FlashAttention-style **online softmax** recurrence logic across stages and heads.
- Hard-coding target-specific workarounds (Metal half-sized tiles, register-pressure caps) that leak into the DSL builder.

**Concrete examples:**

- **dsa_splitk_indexer_loss.py:208-300 (make_dsa_splitk_stage1_kernel)**: The entire inner `sk_tile` loop (Q/K staging via manual `sq_idx = sq_block_id * BLOCK_SQ + i`, double `T.Parallel` loads with `if (sq_idx < ASq) && (d_idx < AD)`, `T.gemm`, then massive manual causal+sparse mask + online-softmax update with `row_max_local`/`row_sum_local`, `m_i_prev`, exp renormalize, and *identical* head-0 `idx_scores_f` path). This is ~120 lines of imperative index math inside what should be a clean tile dataflow. Compare to the cleaner `fp8_amax.py` style or TileLang's own tile-library philosophy (tiles own movement/pipelining). Leaks Triton reference implementation details (`tl.load(..., mask=...)` → predicate `if valid`).

- **dsa_splitk_indexer_loss.py:402-520 (make_dsa_splitk_stage2_kernel)**: Even worse duplication—recomputes Q@K^T *per-head* (full `for h in T.serial(AH)` with identical staging/gemm/mask/exp logic as stage1), then manual `softmax_attn` accumulation + `INV_AH` average + index-softmax + KL term. Online stats (`m_h`/`d_h`, `m1_local`/`d1_local`) are reloaded per-tile. Violates the "single TileLang source for both targets" claim because Metal register pressure forces tiny blocks while CUDA uses large ones.

- **fp8_vecmat_path_c.py:300-400 (make_fp8_vecmat_reduce_kernel)**: Three branches (`vectorized_loads`, `_uses_fp8_dot4_packed_macro`, scalar fallback) with duplicated accum/reduce logic. The packed-dot4 path calls a custom intrinsic; others fall back to scalar casts. This is a **leaky abstraction**—the DSL should hide decode/reduce variants behind `T.fp8_scaled_matmul` or a tile primitive, not Python `if` on `vec`/`K`. Z3 wiring (PassConfig probe + lock) is good but buried in module globals.

- **topk_selector.py:600-750 (_path_c_kernel_for)**: Manual local mini-heap insertion (`for p in T.serial(1, _TOPK_C_K)` with shifts), shared `pair_vals`/`pair_idx` tree merge with `stride` powers-of-two, and post-rewrite `_path_c_rewrite_merge_round` (string surgery on generated MSL at lines ~720-740). Violates TileLang's dataflow-centric goal; the rewrite is a brittle post-processing hack to fix TileLang's conservative barriers (see Z3-final note below).

**Architectural smell**: Each module reinvents "target-aware block constants" (`_metal_block_overrides`, `_block_constants_for_target`, `_path_c_threads_for`) and JIT caching (`_stage1_kernel_for`/`_lru_cache`, `_fp8_vecmat_kernel_for`, `_path_c_kernel_for`). This duplicates the pattern from `fp8_amax.py` (simpler) without a shared `TileLangKernelBuilder` base or registry. Result: leaky per-module globals (`_DSA_STAGE1_*`, `_FP8_VM_*`, `_TOPK_C_*`) that pollute the namespace and make extension (new kernels, new targets) painful.<grok:render card_id="5a5a89" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">6</argument></grok:render>

**Simplification suggestion**: Extract a small `cppmega_mlx.nn._tilelang._common` (or `_kernel_builder`) with:
- `@dataclass` for block config per target/stage (reuse `_metal_block_overrides` logic).
- Generic `make_jit_cached_prim(target, make_prim_func, **shape_kw)` that handles lru_cache + scale_bits packing + contiguous enforcement.
- Standard `online_softmax_update(scores_f, m_i, d_i)` helper primitive (or register it as a TileLang extension). This collapses duplication across DSA stage1/stage2 and future kernels. Reduces ~200 LOC of repeated index/softmax code.

### 2. Duplicated Logic & Awkward Coupling
- **Index/bounds handling**: Every kernel repeats `sq_idx = sq_block_id * BLOCK_SQ + i; if (sq_idx < ASq) ...` + `sk_idx` + `in_bounds && (sq_idx >= sk_idx)` causal + sparse `IndexMask` predicate. Same pattern in stage1 (lines 250-270), stage2 (lines 430-450, 480-500), and topk (interval buffers at 520-550). Leaks into Metal/CUDA differences via different BLOCK_*.
- **Sparse vs dense**: `SPARSE = bool(sparse_loss)` + `if SPARSE: s += IndexMask` (or `v +=`) in *both* stages and topk reference. The comment at dsa:140 notes SPARSE_LOSS is *not* fused (host pre-scatter), yet the kernel still branches on it. Coupling to the wrapper in `cppmega/megatron/dsa_splitk_indexer_loss.py` is tight.
- **Z3/PassConfig wiring**: Duplicated across `fp8_vecmat_path_c.py:80-120`, `topk_selector.py:100-140` (candidate dicts, `_filter_supported_pass_configs`, lock-protected cache, env-var override for "NO_Z3"). `dsa_splitk...` and `fp8_amax` lack it entirely (under-engineering for consistency). The `mlx-z3-wiring` fix-rounds (ScopedBVMode, Bind BV clamp, predicate_fusion null guards, etc.) are cited but not uniformly applied; the string-rewrite in topk is a symptom of incomplete upstream discharge.

**Coupling issue**: `dsa_splitk_indexer_loss_tilelang` (lines 650-720) builds `index_mask` on-host with `.scatter_` then passes it *always* (empty when not sparse). The kernel predicates loads anyway. This couples host Python logic tightly to kernel internals; a better extension point would be optional `IndexMask` buffer + constexpr elimination (TileLang supports it via shape specialization).

**Simplification**: Centralize bounds helpers or use TileLang's mask support more aggressively (if available post-Z3). Make sparse a compile-time `constexpr` param that dead-codes the `if SPARSE` branches (already partially done via `SPARSE = bool(...)` inside `@T.prim_func`).

### 3. Naming/Clarity & Over-/Under-Engineering
- **Naming**: `_DSA_STAGE1_BLOCK_*` globals (dsa:50-70) vs `_metal_block_overrides` (lines 90-120) vs `_block_constants_for_target` (line 580). Inconsistent (`_FP8_AMAX_*` in fp8_amax, `_FP8_VM_*` in vecmat). `dsa_splitk_path_c_status` vs `fp8_amax_path_c_status` vs `topk_selector_path_c_status`/`path_b_status`—good but the `PathCStatus` dataclass in topk duplicates the pattern from fp8_amax/dsa without reuse.
- **Clarity**: Heavy use of `cast(Any, T)` (dsa:170, fp8_amax:150) and `T = cast(Any, T)` to silence pyright. The docstring at dsa:1-140 is excellent but the code inside `make_*_kernel` is dense imperative. Metal overrides comment (dsa:100-130) explains register pressure well but the 32x32x16 choice is buried.
- **Over-engineering**: Topk's `_path_c_rewrite_merge_round` (string regex replace on lowered MSL body) and `canonical_vecmat_runtime_body` (lines 480-510) for feature counting/blockers. The `fp8_vecmat_msl_blockers` / `features` dicts are diagnostic but indicate the Path C lowering isn't yet matching Path B's hand-tuned MSL (packed uint32 loads, simd_sum, lut_dot4). Under-engineering for vecmat: three branches instead of a single `T.fp8_scaled_vecmat` primitive.
- **Z3 conservatism**: Excellent (defaults OFF, lock on caches, filter unsupported keys, env NO_Z3). But `dsa` and `fp8_amax` miss the wiring—apply uniformly or factor into `_tilelang_common`. The "gb10 build drifts" fix-round mentions (Z3 22 ambig, cmath, etc.) suggest upstream TileLang/TVM stability issues that the per-module probes already mitigate.

**Simplification**: 
- Consolidate status dataclasses and `_tilelang_available` into one helper.
- For DSA: Consider collapsing stage1+stage2 into a single fused PrimFunc if TileLang's pipelining/shared budget allows (or keep two-stage but share more staging logic). The upstream Megatron PR #4039 two-stage is the reference, but for Metal the tiny blocks make fusion more attractive.
- Remove or upstream the MSL string rewrites (topk rewrite, vecmat blockers). Rely more on Z3 passes (predicate_fusion, bound checks, simd_lift) after the cited fix-rounds.

### 4. Performance & Portability Suggestions (Concrete)
- **Metal register pressure** (dsa:100-130 comment): The half-size override is pragmatic but couples block choice to kernel structure. TileLang's tile recommendation/inference (per docs) could help automate this; currently manual. Test with larger Metal blocks post-Z3 (simd_lift_reductions, drop_provable_bound_checks) to reduce spilling.<grok:render card_id="f493b5" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">13</argument></grok:render>
- **Pipelining & double-buffering**: DSA stage1 uses `T.Pipelined(SK_TILES, num_stages=...)` with shared Q/K (good), but manual init/zeroing of fragments before each tile (lines 230, 410) adds overhead. Leverage `T.clear` more consistently and let the scheduler handle prologue/epilogue.
- **Topk**: The Path C tree-merge is correct but the local insertion loop (lines 640-660) with early `break` for small K is a manual optimization. The Path B direct-MSL is kept for speed; consider deprecating one path once Path C + Z3 matches (current blockers dict at fp8_vecmat:550 shows the gap).
- **General**: Use `T.gemm` more aggressively in DSA stage2 (currently per-head scalar accum after gemm). For sparse, consider fusing the mask generation if TileLang scatter support lands.
- **Caches & threading**: Good use of `lru_cache` + lock in mlx-z3-wiring. Ensure all `_*_kernel_for` functions are thread-safe (already mostly are).

### Round-Final Summary (Post-Fix-Rounds)
The architecture successfully delivers **unified CUDA/Metal lowering** with conservative Z3 opt-in (PassConfig + locks, env gates, filtering)—a strong foundation matching the "mlx-z3-wiring" intent and TileLang's dataflow-centric philosophy. However, it is **overly imperative and leaky**: the `@T.prim_func` bodies read like lowered Triton/CUDA rather than high-level tile dataflow, with duplicated bounds/online-softmax/index logic and target-specific hacks. This increases maintenance cost for new kernels or TileLang upgrades and risks subtle Metal/CUDA divergence (register pressure, barriers).

**Prioritized simplifications** (actionable, reduce LOC/complexity):
1. **Extract common builder** (new `_tilelang_common.py`): shared status helpers, block config dataclass, JIT wrapper, online-softmax primitive, bounds helpers. Apply Z3 wiring uniformly. (Eliminates duplication across 4 files.)
2. **Raise abstraction in DSA kernels**: Factor Q@K^T + mask + online update into reusable tile loops or sub-functions inside the PrimFunc (TileLang supports nesting). Collapse head-0 special case if possible or make it a template param.
3. **Clean vecmat/topk**: Converge Path C toward a single `T.*` primitive path; deprecate string rewrites once Z3 (predicate_fusion, bound checks, simd_lift) + fix-rounds (null guards, BV clamp, etc.) discharge the needed patterns. Use TileLang tile library where available.
4. **Naming/extension**: Standardize `*_path_c_status` + block resolution. Add explicit extension hook (e.g., `register_custom_prim`) for future fused variants (backward, fp8-indexer, sparse-MLA).

These changes simplify without losing portability or the conservative Z3 stance. The modules are already production-oriented (lru_cache, contiguous enforcement, shape validation, detailed docs); tightening the DSL usage will make them more maintainable and closer to TileLang's intended "focus on algorithm, compiler handles scheduling" model.<grok:render card_id="9b1ae0" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">13</argument></grok:render>

No new bugs found that would cause incorrect numerics (bounds predicates and sentinels match Triton reference), but the imperative style increases risk of off-by-one on boundary tiles or Metal-specific races (shared init without explicit barriers in some paths—ensure `T.sync_threads()` after writes). Performance on Metal is gated by tiny blocks; Z3 progress should help close the gap to Path B.
