**Wave-3 review summary (tip 5b83522 + fix-round-3 6b74ec7+5b83522)**

The topk insertion-sort early-break fix is **correct** (standard `else: break` insertion into ascending min-heap of top-K; the prior `elif (K<=8) or (K>=64): break` heuristic was the source of the K=16/32 corruption). The "1.5-1.94× speedup" on K=32 was illusory — it produced wrong outputs. Current Path C is ~1.28× slower than Path B at that point, which is why `_PATH_C_AUTO_PROFITABLE_RECEIPTS` correctly dropped `(1, 512, 32, "float32")` and `backend="auto"` now prefers Path B there. No evidence of other "fast because broken" cases in the bundle (sparse_mla Path C ratios look clean; the MSL canonicalize/hoist fix in `_msl_transform` did not introduce compile issues per the bench JSON).

Focus is strictly on **design/architecture** (violated abstractions, leaky modules, duplication, coupling, missing extension points, naming/clarity, over/under-engineering). I prefer simplifications.

### 1. Leaky abstraction & duplicated lowering logic (`topk_selector.py`)

The module mixes three very different implementation strategies without a clean seam:

- **Reference** (pure MLX, lines 233–336)
- **Path B** (direct handwritten MSL via `_msl_transform.make_metal_kernel`, lines 344–689)
- **Path C** (TileLang `@T.prim_func` → `_msl_transform.lower_tilelang_to_msl_inline` + post-processing, lines 695–1135)

This leads to **massive duplication** of shape/dtype/masking/validation logic:

- `_path_c_score_dtype`, `_path_c_interval_buffers`, `_path_c_default_interval_buffers` (Path C only) vs. identical logic inside `topk_selector_metal` (lines 640–661) and `topk_selector_reference` (lines 275–312).
- `has_starts` / `has_ends` template flags + sentinel handling duplicated in MSL source (lines 390–401) and in the Python reference.
- Threadgroup budget calculations duplicated in spirit (`_path_c_threads_for` vs. the `max_pair_slots` / power-of-2 logic in `topk_selector_metal:628–637`).

**Concrete issues:**

- `topk_selector.py:1161` (`topk_selector_path_b_status`) and `topk_selector_path_c_status` duplicate the Metal availability check already present in `_msl_transform.msl_dispatch_status`.
- The public `topk_selector(..., backend=...)` (lines 1178–1224) is a giant dispatch table that re-implements routing knowledge already encoded in `_path_c_auto_profitable` + receipt set. This is classic leaky module: callers (and tests) must know the internal receipt rules.
- `_path_c_rewrite_merge_round` (lines 845–916) is a **fragile textual post-processing hack** on the MSL body produced by `_msl_transform.lower_tilelang_to_msl_inline`. It hard-codes regexes against TileLang's emission (lane expr, barrier patterns, cast expressions). The TODO you left at the top of the file acknowledges the fragility, but the design itself is the problem: you're fighting the abstraction instead of extending the lowering pipeline.

**Suggestion (simplify):** Introduce a narrow `TopKStrategy` protocol / abstract base with three concrete impls (`ReferenceStrategy`, `MetalDirectStrategy`, `TileLangStrategy`). Each strategy owns its own dispatch status, kernel construction, and input canonicalization (`_normalize_inputs` that produces `(work_scores, starts_buf, ends_buf)`). The public function becomes a thin router that tries strategies in priority order (auto = receipt-driven Path C → Path B → ref). This collapses the duplicated masking/buffer logic into one place and makes `_path_c_rewrite_merge_round` an internal detail of `TileLangStrategy` (or better, a proper TIR pass once TileLang exposes hooks).

### 2. Over-engineering in `_msl_transform.py` vs. under-engineering in topk

`_msl_transform.py` is a **general-purpose** TileLang → inline-MSL adapter (prelude/body splitting, builtin alias canonicalization, Z3 `pass_configs` threading, FP8 intrinsic registration, libz3/dlopen hacks, etc.). That's good for the sparse_mla / fp8_amax / dsa kernels.

But the **topk** Path C uses it in a very specific way (shape-specialized kernel via globals mutation + `@lru_cache` on `_path_c_kernel_for`) and then immediately does another layer of string surgery (`_path_c_rewrite_merge_round`). This creates awkward coupling:

- `topk_selector.py:1058` calls the general lowerer, then immediately mutates the result.
- Globals mutation pattern (`globals().update(T=T, _TOPK_C_*=...)` at lines 939–947) before defining the `@T.prim_func` is brittle and surprising (especially with the static placeholder comments at 221–226).

The `_TOPK_C_*` globals + lru_cache on a 5-tuple key works, but it leaks the macro style into the public module and makes the TileLang DSL feel second-class compared to the clean handwritten MSL in Path B.

**Suggestion (simplify):** For topk (a relatively simple kernel), consider making the TileLang Path C a **thin wrapper** around the same static-threadgroup pattern that made Path B successful. Or go the other way: make the direct-MSL Path B the *only* fast path and keep TileLang Path C purely for verification/correctness coverage on CUDA (where it may still be competitive). The current dual-Path design with different lowering pipelines for essentially the same algorithm (local insertion + tree merge) is expensive in maintenance.

### 3. Naming & clarity problems

- `Path B` / `Path C` terminology is internal jargon that leaks everywhere (`topk_selector_path_b_status`, `_PATH_B_OK_REASON`, bench JSONs, test names). It makes the module harder to read for new contributors. Prefer `direct_msl` / `tilelang_dsl` or `vendor_msl` / `tiledsl`.
- `_path_c_rewrite_merge_round` (845) — the name doesn't reveal it's a **barrier elision + cast tightening** hack. Call it what it is: `_tighten_tilelang_merge_round_barriers` or similar.
- `_msl_transform.py:474` `_inline_tilelang_kernel_body` injects the `uint3 blockIdx/threadIdx` aliases and then immediately canonicalizes them away in most cases. The round-trip is confusing.
- In `dsa_splitk_indexer_loss.py` (the truncated file), you mention block-size overrides for Metal vs CUDA. That file (not fully shown) seems to repeat similar "Path C vs reference" patterns — watch for the same duplication creeping in.

### 4. Missing extension points / rigid policies

- The strict/auto routing policy lives in two places: the JSON bench artifacts (`strict_policy`, `auto_routing_policy` with hardcoded receipts) and the Python code (`_PATH_C_AUTO_PROFITABLE_RECEIPTS` at 761–776 + `_path_c_auto_profitable`). This is a classic source-of-truth problem. A single source (e.g. a small policy dataclass loaded from JSON or a registry) would be better.
- No easy way to plug in a new strategy (e.g. a future "Path D: even better handwritten MSL" or a CUDA Triton fallback). The dispatch table in `topk_selector` is hardcoded.
- Z3/pass_configs support is wired only into the general lowerer and selectively into topk (`_topk_path_c_pass_configs`). The comment at the top of `_msl_transform` talks about a "Z3 roadmap" — the current design makes it hard to know which kernels actually benefit from which passes without reading the call sites.

**Simplification:** A `KernelRegistry` or simple strategy selector that registers strategies with their capabilities (supports_masking, profitable_shapes, lowering_type) would collapse the `backend=` if-ladder and the receipt set into declarative data.

### 5. Minor but symptomatic issues

- `topk_selector.py:1169` — `topk_selector_path_b_status` checks `_TOPK_KERNEL is None` but the construction of `_TOPK_KERNEL` (584) can fail silently in some environments. The status should probably call the same `msl_dispatch_status` path used elsewhere.
- Heavy use of `lru_cache(maxsize=128)` on kernel construction is fine for perf but couples caching policy to shape tuples. If you ever need to invalidate (e.g. after a TileLang rebuild), it becomes painful.
- The MSL source in Path B is a huge string literal (344–575) with many commented-out "we tried X but memory didn't fit" explanations. Good for history, but it makes the file long and the active code harder to see. Consider extracting to a separate `.metal` file and reading it at module level (or keep the comment but split the implementation into smaller helper functions).

### Overall architectural recommendation (simplify)

The current design treats "Path C = TileLang DSL" as a first-class peer to "Path B = handwritten MSL", but the reality after fix-round-3 is that **Path B is the robust, performant production path** on Metal, while Path C is primarily a correctness/verification vehicle (and a way to share code with CUDA). Lean into that asymmetry instead of maintaining two parallel lowering pipelines with duplicated input handling and fragile post-processing.

**Concrete next steps (in order of impact):**

1. Extract common input normalization/masking into a single `_prepare_topk_inputs(scores, k, starts, ends) -> tuple[mx.array, mx.array, mx.array]` used by reference + metal + tilelang.
2. Collapse the three strategies behind a small `TopKBackend` enum + registry so `topk_selector` becomes ~20 lines of "try in priority order".
3. Move the fragile `_path_c_rewrite_merge_round` (and the globals mutation pattern) deeper into the TileLang strategy or replace the TileLang Path C with a simpler static MSL variant that reuses the same merge logic as Path B (DRY win).
4. Make routing policy data-driven (one source of truth for "these shapes are green for Path C").

This would reduce the module's cognitive load significantly, make the "fix-round-3" correctness win clearer (no more magical 1.9× that was actually wrong), and leave the general `_msl_transform` machinery focused on the kernels that truly need the full TileLang+Z3 pipeline (sparse_mla, fp8_amax, dsa_splitk, etc.).

The sparse_mla bench JSON looks healthy post-canonicalize fix; no new red flags there. The topk tests are thorough and correctly switched to set-equality where order is unspecified — good.
