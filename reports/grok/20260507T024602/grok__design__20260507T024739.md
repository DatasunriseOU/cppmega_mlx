**Overall Architecture Assessment (Wave-2 post fix-round-2)**

The three modules (`dsa_splitk_indexer_loss.py`, `fp8_amax.py`, `fp8_vecmat_path_c.py`, `topk_selector.py`) follow a consistent **Path A/B/C** pattern for unified CUDA/Metal support via TileLang DSL (Path C) + direct-MSL bypass (Path B) + reference fallbacks. This is a solid, pragmatic design for a performance-critical MLX/C++Mega integration. The modules share good patterns: status dataclasses, `_tilelang_available()`, target resolution, lru_cache'd kernel builders, and Metal-specific block/thread tuning with budget comments.

However, **design issues** persist around **leaky abstractions**, **duplicated boilerplate**, **over-engineering in places**, and **coupling** to TileLang internals/MLX lowering details. The recent commit 9098d6a (M_pre/D_pre budget gate, topk bounds/NaN guards) correctly addresses prior register-spill and safety gaps on Metal, but introduces minor new awkwardness. The code is now more robust but still carries complexity debt from supporting two very different backends (CUDA Triton-style vs. Apple SIMDgroup Metal) through a single TileLang DSL surface.

### 1. **Violated Abstractions / Leaky Modules** (Most Impactful)

- **dsa_splitk_indexer_loss.py:120-280 (`_metal_block_overrides` + wave comments)**: The function hardcodes deep knowledge of *register pressure* (M_pre/D_pre fragments `AH * BLOCK_SQ * 4` bytes, stage-2 score fragments, shared Q/K staging) and Metal's 32 KiB threadgroup limit. This leaks low-level Metal/TileLang codegen details into the high-level kernel builder. The comment block is excellent documentation, but the logic (AH > 64 → halve BLOCK_SQ) belongs in a dedicated `MetalResourceBudget` helper or TileLang pass, not scattered across stage overrides.<grok:render card_id="8b5d45" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">1</argument></grok:render><grok:render card_id="7aae32" card_type="citation_card" type="render_inline_citation"><argument name="citation_id">2</argument></grok:render>

  **Actionable**: Extract to a new dataclass or `_metal_budget` module-level helper that returns `(BLOCK_*, threads, num_stages, use_md_pre: bool)`. Call it once in `_block_constants_for_target` and document the *policy* ("keep AH*BLOCK_SQ*8 <= 16 KiB for pre-load pair"). This makes the gate reusable and the stage-1/2 split less magical. The wave-2 `USE_MD_PRE` fallback (HBM re-read when over-budget) is correct but currently duplicated in the (truncated) stage-2 kernel builder.

- **topk_selector.py: ~400-600 (Path C merge rewrite + _path_c_rewrite_merge_round)**: Heavy post-lowering textual MSL surgery (`_path_c_rewrite_merge_round` with regexes on `thread_position_in_threadgroup.x`, barriers, and address calculations). This is a classic **leaky abstraction** — the TileLang DSL promise is broken by string munging to fix conservative barriers emitted by the lowering. The recent rewrite improves it, but it remains fragile to TileLang changes.

  **Better design**: Push the active-lane optimization (`if ((tid & ((stride*2)-1)) == 0)`) and barrier elision upstream into TileLang/Z3 (via the existing PassConfig machinery already wired for "tl.drop_provable_bound_checks"). If not feasible, encapsulate the rewrite in `_msl_transform` (shared with `fp8_vecmat_path_c.py`) rather than duplicating rewrite logic here.

- **Cross-module duplication**: 
  - `_tilelang_available()`, status dataclasses (`DSASplitKPathCStatus`, `FP8AmaxPathCStatus`, `FP8VecmatPathCStatus`, `PathCStatus`), `_resolve_target`, `_is_metal`, `tilelang_supports*` variants.
  - Metal target string handling (`"metal -thread_warp_size=32"`, `_as_metal_target` in vecmat).
  - PassConfig filtering/caching (`_fp8_vecmat_pass_configs`, `_topk_path_c_pass_configs`, first-run locks).
  - Shape bucketing / cache keys in fp8_amax (power-of-two) vs. exact-shape in others.

  **Suggestion**: Introduce a small `_tilelang_common.py` (or expand `cppmega_mlx.nn._tilelang`) with shared helpers: `TileLangStatus`, `make_status`, `resolve_target(device)`, `pass_config_cache_for(candidates)`, `metal_block_overrides_for_budget(...)`. This reduces copy-paste and makes "add a new Path C kernel" a matter of subclassing/extending rather than reimplementing the scaffolding. Current design makes each file ~self-contained (good for isolation) but violates DRY at the architecture level.

### 2. **Awkward Coupling & Missing Extension Points**

- **dsa_splitk_indexer_loss.py: ~800+ (make_dsa_splitk_stage* kernels)**: The two-stage split (online-softmax stats in stage 1, KL recompute in stage 2) mirrors the upstream Triton/Megatron PR exactly. This is correct for numerical equivalence, but the kernels are tightly coupled through intermediate buffers (`softmax_m`, `softmax_d`, `M1/D1`). No clear "fused single-kernel" extension point for future work (mentioned as deferred). The `sparse_loss` path (pre-computed IndexMask on host) and `pg_collection` (ignored TP) are handled at wrapper level — good, but the PrimFunc signatures are stabilized at the cost of always passing `IndexMask` (even empty).

  The wave-2 `USE_MD_PRE` conditional alloc + fallback is a good pragmatic fix, but it lives inside the truncated stage-2 builder and uses magic constants. Consider a small `StageConfig` dataclass passed to the builders.

- **fp8_vecmat_path_c.py & topk_selector.py**: Both have dual Path B (direct MSL) + Path C (TileLang DSL → lower → mx.fast.metal_kernel). The `fp8_scaled_vecmat_path_c` and `topk_selector_tilelang` entrypoints have complex dispatch with `can_run_metal()`, intrinsics checks, simplify fallbacks, and shape canonicalization. Coupling to `_msl_transform` is high (lowering, dispatch, grid/threadgroup computation). 

  **Clarity issue**: `backend="auto"` in topk_selector prefers Path C only on a tiny hardcoded `_PATH_C_AUTO_PROFITABLE_RECEIPTS` set. This is honest (avoids dishonest routing) but makes the "auto" policy opaque and non-extensible. Move the receipt set + profitability heuristic to a configurable policy object.

- **fp8_amax.py**: Cleanest of the set. The amax + quantize split and bucketing (`_bucket_n`) are well-designed. Minor: `_pick_block_size` and `_target_family` duplicate some target logic present elsewhere.

### 3. **Naming / Clarity Problems**

- Inconsistent status naming: `dsa_splitk_path_c_status`, `fp8_amax_path_c_status`, `fp8_vecmat_path_c_status`, `topk_selector_path_c_status` / `path_b_status`. The `PathCStatus` dataclass in topk is too generic.
- `_metal_block_overrides(stage, AH=None)` — "overrides" is vague; it's really `metal_tuned_blocks_for_stage_and_budget`.
- Many "Wave-2 / grok perf #N / grok finding #X" comments — useful for history but clutter the runtime code. Consider extracting a DESIGN.md or ARCH_NOTES per module (or one central tilelang_ports/) and linking with short references.
- `dsa_splitk_indexer_loss_tilelang` wrapper accepts `pg_collection` for API compatibility but ignores it — document or remove if TP fusion is never coming to Path C.

### 4. **Performance & Over/Under-Engineering**

- **Positive (post wave-2)**: The M_pre/D_pre gate + HBM fallback in DSA stage 2 prevents register spill on large AH. Topk bounds-check + NaN guard for fully-masked rows fixes real correctness/safety issues without noticeable overhead (host-side scatter is already there for sparse). Q hoist in stage 1 and partial hoist in stage 2 are good.
- **Remaining**: DSA kernels still have per-sk_tile Q reloads in stage 2 (partial hoist noted as TODO). Full hoist is blocked by accumulator semantics — consider materializing per-tile softmax_attn as a small buffer or restructuring the KL reduction if profiling shows HBM pressure.
- Topk Path C still uses a simple insertion + tree-merge; the original CUDA radix-select was more sophisticated but blocked on Metal shared.dyn. The current design trades some peak perf for portability — acceptable, but the `PAIR_BUF` cap and power-of-two threads are tuned specifically for 32 KiB. Document the exact budget math centrally.
- Over-engineering in vecmat: multiple branches (vectorized_loads probe, dot4 macro, scalar fallback) + simplify fallback + feature counters for tests. The canonical fast path (packed dot4) is good; the probes add maintenance burden.

**Simplification Opportunities**:
1. Centralize Metal budget logic and PassConfig caching.
2. Make Path B/C dispatch a small strategy/policy class instead of if-ladders in each entrypoint.
3. For DSA: expose a `make_dsa_splitk_fused_kernel` builder that composes stage1+stage2 internally if a single-launch fused path becomes viable later (extension point).
4. Reduce textual MSL rewrites by investing more in TileLang/Z3 passes (the existing PassConfig wiring is the right direction).

### 5. **Specific New/Remaining Issues Post 9098d6a**

- **dsa_splitk_indexer_loss.py: ~250-270 (`_metal_block_overrides` for stage=2)**: The AH > 64 logic is now correct, but `USE_MD_PRE` decision (mentioned in comments) is not yet reflected in the provided `_block_constants_for_target` or kernel call sites in the same way as the fragment allocs. Ensure the stage-2 kernel builder reads the same `USE_MD_PRE` flag that the comment describes.
- Topk bounds-check/NaN guard (lines ~ in wrapper): Excellent safety addition. Minor: the `_row_has_valid` patch sets slot 0 to 0 for all-masked rows — verify this doesn't interact badly with causal mask in the kernel (it should be fine).
- No new major regressions spotted in the provided snippets; the fixes integrate cleanly.

**Prioritize**:
- Extract common TileLang/Metal scaffolding to reduce duplication.
- Centralize budget-aware tuning (especially Metal 32 KiB + register pressure).
- Minimize post-lowering string hacks; push more to TileLang passes.

The design is evolving in the right direction (unified source, target-specific tuning, safety guards). With a bit more refactoring toward shared helpers and clearer extension points, it will be much easier to maintain and extend to new kernels (e.g., fused backward, fp8 variants).
