**PLANNING-vs-REALITY AUDIT + DESIGN REVIEW**  
**cppmega.mlx upstream PR program** (review_bundle.md bundle)

Ranked by severity (1 = highest impact on correctness, maintainability, production readiness). Each finding is tagged **(a)–(f)** per your spec, cites exact `<filename>:<line-range>`, and focuses on **design/architecture** (leaky abstractions, coupling, duplication, missing extension points, naming/clarity, over/under-engineering). I prioritized simplifications that remove workarounds, stale docs, and brittle string transforms.

### 1. **(e) UNCOMMITTED-INCOHERENT** — Path C dispatch is fundamentally broken (highest severity)
`bench/tilelang_ports/fp8_path_c_vs_path_b.json:166-171`, `sparse_mla.json:350-362`, `scripts/bench_tilelang_fp8_path_c.py:783`, `cppmega_mlx/nn/_tilelang/fp8_vecmat_path_c.py: (all lowering sites)`  
`AttributeError: Check failed: (reg != nullptr) is false: Operator tirx.metal.fp8_e4m3_dot4 is not registered` appears in **every** Path C bench receipt and test failure.  
**Design issue**: `_msl_transform.py` + `fp8_vecmat_path_c.py` assume a registered Metal intrinsic that the local TileLang/TVM build no longer provides (post-filing of #2144/#2145). This is not a “probe failure” — it is a **leaky abstraction** where the DSL lowering + post-processing layer silently produces un-runnable kernels. Uncommitted `M` status on `_msl_transform.py`, `fp8_vecmat_path_c.py`, and bench scripts (git status in JSON:69) means production routing is built on vaporware.  
**Simplify**: Remove the entire FP8 Path C macro path until the intrinsic is registered **upstream** (or delete the custom op and route everything to Path B scalar fallback). One source of truth, no string transforms.

### 2. **(c) TRACKER-STALE** — Blockers tracker claims victory on FP8 while reality is red
`_path_c_blockers_tracker.md:155-169` (standalone FP8 parity “closed locally”), `:299-305` (Patch B as “tombstone” algebraic rewrite), `:104-109` (Sparse-MLA FP8 composition)  
vs. `_filed_prs_2026_05_04.md:130-131` (explicitly calls #2146 a “tombstone”) and the bench JSON failures above.  
**Design issue**: The tracker became a living document that no longer reflects the **runtime contract** of the code. This violates the single-source-of-truth principle for Path B/C routing.  
**Simplify**: Make `_path_c_blockers_tracker.md` a generated artifact from the bench JSONs (or delete it and embed the strict gates directly in `production_kernel_routing.md`).

### 3. **(a) PLANNED-BUT-MISSING** — _pr_filing_pack.md master plan is obsolete
`_pr_filing_pack.md:1-40` (only documents initial 4 PRs + reality-check note on stacking)  
vs. actual directories present (`tilelang_metal_inline_kernel_body/`, `tilelang_metal_emit_metal_builtins/`, `tilelang_metal_mamba3/`, etc.) and `_filed_prs_2026_05_04.md:5-121` (14 filed PRs).  
`tilelang_metal_inline_kernel_body/README.md:19-24` has only a `.metal` probe — **no 0001-*.patch**. Same pattern in `tilelang_metal_mamba3/`.  
**Design issue**: The filing pack is no longer the authoritative “everything needed to file” artifact. This creates a maintenance nightmare for anyone trying to reproduce the upstream stack.  
**Simplify**: Deprecate `_pr_filing_pack.md` entirely; redirect it to `_filed_prs_2026_05_04.md` + a new “current-stack-apply-order.md” generated from git.

### 4. **(e) UNCOMMITTED-INCOHERENT** — MSL post-processing is the new core abstraction (leaky module)
`cppmega_mlx/nn/_tilelang/_msl_transform.py:7-16` (imports), `:110-128` (fp8_dot4 rewrite test), `:13-15` (docstring admitting Apple MSL forbids threadgroup in non-kernel functions)  
+ `tests/test_tilelang_msl_transform.py:19-142` (exhaustive regex tests for inlining, builtin aliases, bounds-check removal).  
**Design issue**: String-based canonicalization of generated MSL is now the **real** Metal backend. It is tightly coupled to TileLang’s exact codegen output format, has no extension points for new ops/intrinsics, and duplicates logic that should live in TileLang’s `codegen_metal.cc`. The workaround was supposed to be temporary (see followup candidates) but became permanent.  
**Simplify**: Upstream the two remaining transforms (#1 inline kernel body was already “closed false-alarm” per ac80716) and delete `_msl_transform.py` entirely. One fewer layer, zero regex fragility.

### 5. **(f) CROSS-REF** — production_kernel_routing.md no longer matches reality
`docs/production_kernel_routing.md` (M in git status) vs. `bench/tilelang_ports/fp8_path_c_vs_path_b.json:321-340` (Path C vecmat dispatch fails) and `sparse_mla.json:34-37` (“Path C ready” but all strict gates red).  
**Design issue**: The AUTO routing table and strict gates are documented against a world that no longer exists. This is classic documentation-as-code smell — the JSON receipts are the real source of truth, but the Markdown is hand-maintained.  
**Simplify**: Generate the routing table from the bench JSONs at build time (or make the JSON the single source and delete the Markdown table).

### 6. **(d) FOLLOWUP-CANDIDATES** — _path_c_followup_pr_candidates.md is frozen in time
`_path_c_followup_pr_candidates.md:17-60` (Candidate #1 inline_kernel_body “ready-to-write”, #2 builtins)  
vs. actual filings #2143/#2147 and ac80716 closure. No update to the doc.  
**Design issue**: Follow-up candidates became another stale planning artifact instead of a living roadmap. No clear “filed / superseded / closed” state machine.  
**Simplify**: Convert the file to a simple checklist table with columns “Candidate | PR # | Status | Next action” (one source of truth).

### 7. **(b) FILED-BUT-UNDOCUMENTED** — _filed_prs_2026_05_04.md is the only accurate history
`_filed_prs_2026_05_04.md:123-140` correctly labels #2146 as tombstone and #2147 as partial, but `_pr_filing_pack.md` and `local_build_status.md:19-32` still pretend the original 4-PR pack is the full story.  
**Design issue**: Multiple overlapping “history” documents with no single owner. Classic duplicated logic at the documentation layer.  
**Simplify**: Retire `_pr_filing_pack.md` and `local_build_status.md`; keep only `_filed_prs_2026_05_04.md` + a one-line redirect.

### 8. **Design** — Duplicated FP8 lowering logic (Path B vs Path C)
`cppmega_mlx/nn/_tilelang/fp8_vecmat_path_c.py` (entire file, especially lowering template) + Path B hand-written MSL in the original audiohacking kernels.  
**Issue**: Identical packed `uint32` + LUT `dot4` + scale-after-accum logic lives in two places. No shared abstraction (TileLang op, scheduler, or common MSL macro).  
**Simplify**: Make Path C call the **same** Metal kernel builder that Path B uses (extract into `mx.fast.metal_kernel` helper or a new `_tilelang/fp8_common.py`).

### 9. **Naming / clarity** — “Path C” is overloaded and misleading
Every file uses “Path C” for: (1) TileLang DSL, (2) lowered MSL + postproc, (3) AUTO dispatch gate.  
See `_path_c_blockers_tracker.md:20-60`, `production_kernel_routing.md`, `fp8_vecmat_path_c.py:1-10`.  
**Issue**: Violates the “one concept, one name” rule and makes the routing table incomprehensible.  
**Simplify**: Rename to `tilelang_path` everywhere (or `dsl_path`) and keep “Path B = direct MSL”.

### 10. **(e) UNCOMMITTED-INCOHERENT** — Orphaned test vs production mismatch
`tests/test_tilelang_msl_transform.py:110-128` (fp8_vecmat_dot4 rewrite test) vs. actual call sites in `fp8_vecmat_path_c.py` (many dropped after recent refactors).  
**Issue**: Tests cover code paths that no longer exist in production. Classic test-code drift from uncommitted half-finished refactors.  
**Simplify**: Delete the orphaned tests or make them integration tests against the real dispatch path.

**Overall architecture recommendation (simplest path forward)**  
1. Fix the FP8 intrinsic registration (root cause of all dispatch failures).  
2. Delete `_msl_transform.py` + all string post-processing once the two remaining upstream fixes land.  
3. Collapse Path B/C into a single `tilelang_path` with clear strict gates sourced from JSON receipts.  
4. Retire the three stale planning documents; keep one canonical `_filed_prs_*.md` + generated bench receipts.  

This removes ~400 LOC of fragile string transforms, eliminates doc drift, and restores a clean “DSL → correct Metal” contract. The current design has turned a temporary workaround layer into the de-facto Metal backend — exactly the opposite of the original upstream-PR goal.
