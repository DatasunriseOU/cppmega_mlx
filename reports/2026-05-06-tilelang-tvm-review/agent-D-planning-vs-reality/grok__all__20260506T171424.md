### 1. GROUPED FINDINGS

#### **Correctness (highest severity)**
- FP8 Path C is **completely broken at runtime** despite “parity closed” claims in the trackers.  
  `bench/tilelang_ports/fp8_path_c_vs_path_b.json:167` (and `:333` for vecmat), `sparse_mla.json:351`–`362` (and `388`–`405` for e8m0):  
  `AttributeError: Check failed: (reg != nullptr) is false: Operator tirx.metal.fp8_e4m3_dot4 is not registered`  
  Same error surfaces in `fp8_vecmat_path_c.py` lowering and `sparse_mla_path_c` QK reducer. The intrinsic was supposed to be supplied by PRs #2144/#2145 (storage+vector) + #2146, but the current TileLang build (`/tmp/tl_apache_tvm_swap`) does not have it. Path C therefore never dispatches; benches fall back or fail outright.
- `test_tilelang_fp8_vecmat_path_c.py:217`–`222` (and similar in bench harness) expect `fp8_scaled_vecmat_path_c` to succeed or skip gracefully; the real dispatch path raises before the test even reaches the parity assertion.
- `_msl_transform.py` FP8 dot4 rewrite pattern (tested at `test_tilelang_msl_transform.py:110`–`128`) assumes the intrinsic exists; the transform silently produces invalid MSL when the intrinsic is missing.

#### **Security**
- None material. All string transformations run on trusted, locally-generated TileLang MSL output. No user-controlled input reaches the regexes or `eval` paths. The only potential vector would be a malicious TileLang codegen change, which is out of scope.

#### **Performance**
- **Intended win path is dead**: FP8 vecmat/matmul receipts in older probes claimed 0.82×–0.95× vs Path B, but current benches show `path_c_ok=False` everywhere (`fp8_path_c_vs_path_b.json:321`, `sparse_mla.json:438`). The codebase therefore always falls back to the hand-written Path B MSL kernels, defeating the entire TileLang investment for FP8.
- `_msl_transform.py` performs multiple regex passes (`_inline_tilelang_kernel_body`, `_canonicalize_tilelang_builtin_aliases`, `_rewrite_tilelang_builtin_axis*`, `_remove_redundant_kv_bounds_checks`, etc.) on every generated kernel. While the kernels are tiny, the string manipulation is unnecessary once the upstream candidates land.
- Uncommitted bench JSON variants (`fp8_path_c_vs_path_b.quick.*.json`, `*.live.json`, `*.current_strict.json`) contain stale/hand-edited timing data that no longer match reality.

#### **Design**
- **Heavy downstream MSL post-processing** (`cppmega_mlx/nn/_tilelang/_msl_transform.py`) exactly matches the workarounds described in `_path_c_followup_pr_candidates.md:17`–`62` (Candidates #1 and #2). This is the canonical evidence that the upstream codegen gaps still exist, yet the filed PRs (#2143–#2147) have not yet removed the need for the transform.
- `fp8_vecmat_path_c.py` + `_msl_transform.py` duplicate logic that should live in TileLang’s Metal macro emitter / scheduler. The current design creates a maintenance fork that must be kept in sync with every TileLang Metal change.
- Dispatch policy (`AUTO` in production_kernel_routing + topk_selector) is per-shape receipt-based and fail-closed, which is sound, but the receipts themselves are stale (see bench JSONs).

#### **Test Coverage**
- `test_tilelang_msl_transform.py` is excellent for the string transforms (covers inline-body, builtin aliases, cast variants, comments/strings, FP8 dot4 pattern, etc.).
- `test_tilelang_fp8_vecmat_path_c.py` and `test_tilelang_bench_harness.py` only exercise the happy path or explicit skips; they do **not** assert that the intrinsic registration succeeded or that Path C actually dispatched.
- No integration test forces a full `TileLangMSLLowering → dispatch` round-trip with the real patched TileLang build. Consequently the registration bug was invisible to CI.
